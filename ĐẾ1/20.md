# 1. Dịch câu hỏi

Một công ty phát triển phần mềm đang sử dụng **serverless computing với AWS Lambda** để build và chạy application mà không cần quản lý server.

Công ty có một Lambda function kết nối tới:

* MongoDB Atlas
* một third-party API

Developer được yêu cầu tạo environment variables cho nhiều môi trường:

* DEV
* SIT
* UAT
* PROD

Các biến này gồm:

* MongoDB hostname
* username
* password
* API credentials

Vì Lambda function đang lưu thông tin nhạy cảm như database credentials và API credentials, làm thế nào để bảo vệ các thông tin này, tránh việc developer khác hoặc người khác nhìn thấy plain text?

Chọn giải pháp bảo mật tốt nhất.

---

# 2. Đáp án đúng

✅ **Create a new AWS KMS key and use it to enable encryption helpers that leverage AWS KMS to store and encrypt the sensitive information.**

Dịch:

> Tạo một AWS KMS key mới và dùng key đó để bật encryption helpers, sử dụng AWS KMS để lưu và mã hóa các thông tin nhạy cảm.

# 4. Lambda có mã hóa environment variables mặc định không?

Có.

Lambda mặc định mã hóa environment variables bằng:

# AWS KMS default service key

Nhưng vấn đề là:

```text
Người có quyền xem Lambda configuration vẫn có thể thấy giá trị plain text trong console
```

=> mặc định chưa đủ an toàn cho requirement này.

---

# 5. Encryption helpers là gì?

Encryption helpers cho phép bạn:

```text
mã hóa giá trị secret trước khi lưu vào environment variables
```

Tức là Lambda console chỉ thấy ciphertext.

Ví dụ:

```text
DB_PASSWORD = AQICAHj9x...
```

Không thấy password thật.

Khi Lambda chạy, code sẽ gọi KMS decrypt để lấy giá trị thật.

---

# 6. Flow đúng

```text
Developer
   ↓
Encrypt secret bằng KMS key
   ↓
Lưu ciphertext vào Lambda environment variables
   ↓
Lambda runtime
   ↓
Gọi KMS decrypt
   ↓
Dùng secret để connect MongoDB/API
```

---

# 7. Vì sao phải tạo KMS key mới?

Default KMS key của Lambda:

* AWS managed
* ít kiểm soát hơn
* không dùng tốt với encryption helpers theo requirement
* quyền truy cập thường rộng hơn trong team

Customer managed KMS key cho phép:

* quản lý key policy
* phân quyền ai được decrypt
* rotate key
* disable key
* audit usage
* kiểm soát access tốt hơn

---

# 8. Vì sao đáp án đúng?

Vì nó đáp ứng đủ:

* encrypt sensitive variables
* không hiện plaintext trong console
* dùng KMS
* kiểm soát quyền decrypt
* audit được key usage
* bảo mật cao nhất trong các lựa chọn

---

# 9. Vì sao các đáp án khác sai?

## ❌ "Không cần làm gì vì Lambda đã encrypt mặc định"

Sai vì:

Đúng là Lambda encrypt at rest mặc định.

Nhưng:

```text
người có quyền xem Lambda function vẫn có thể thấy plaintext
```

Đề yêu cầu không để developer khác thấy credentials.

=> cần encryption helpers + KMS key riêng.

---

## ❌ SSL encryption + CloudHSM

SSL chỉ bảo vệ:

```text
data in transit
```

Ví dụ:

```text
Lambda → MongoDB Atlas
```

Nhưng credentials trong Lambda env vẫn có thể bị nhìn thấy.

CloudHSM cũng quá nặng cho case này.

---

## ❌ Lambda không hỗ trợ encryption env variables, chuyển sang EC2

Sai.

Lambda có hỗ trợ encrypt environment variables bằng KMS.

Không cần chuyển sang EC2.

---

# 10. KMS default key vs customer managed key

| Loại key                 | Ý nghĩa                                         |
| ------------------------ | ----------------------------------------------- |
| AWS managed key          | AWS tạo/quản lý cho Lambda                      |
| Customer managed KMS key | Bạn tạo và kiểm soát policy/permission          |
| Encryption helpers       | Mã hóa secret phía client trước khi lưu vào env |
