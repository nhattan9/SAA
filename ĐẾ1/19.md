# 1. Dịch câu hỏi

Một công ty có web application sử dụng Amazon S3 và Amazon CloudFront để phân phối:

* images
* videos
* static contents

cho người dùng toàn cầu.

Công ty vừa thêm tính năng:

# member-only access

cho một số media files chất lượng cao.

Yêu cầu:

* chỉ paying subscribers mới được truy cập private media files
* cần hỗ trợ truy cập nhiều private files
* KHÔNG muốn thay đổi URL hiện tại

Giải pháp nào phù hợp nhất?

---

# 2. Đáp án đúng

✅ **Use Signed Cookies to control who can access the private files in your CloudFront distribution**

Dịch:

> Sử dụng Signed Cookies để kiểm soát ai được phép truy cập private files trong CloudFront distribution.

---

# 3. Keyword quan trọng của đề

---

## Keyword 1

# "private media files"

=> cần:

# access control cho private content

---

## Keyword 2

# "only paying subscribers"

=> cần authorize user.

Không phải public access.

---

## Keyword 3

# "multiple private media files"

Đây là keyword QUAN TRỌNG.

Không phải chỉ 1 file.

Mà là:

# nhiều files

---

## Keyword 4

# "without having to change the current URLs"

Đây là keyword QUAN TRỌNG NHẤT.

=> không muốn generate signed URL mới cho từng file.

=> Signed URL không phù hợp.

---

# 4. Signed URL vs Signed Cookies

Đây là kiến thức cực kỳ hay ra exam.

---

# Signed URL

Dùng khi:

```text id="tx6ew1"
1 URL
1 file cụ thể
```

Ví dụ:

```text id="vywl5z"
download premium.pdf
```

---

# Signed Cookies

Dùng khi:

```text id="dlalxq"
1 user
được access NHIỀU files
```

Ví dụ:

```text id="8y6xma"
premium videos
premium images
subscriber area
```

---

# 5. Vì sao Signed Cookies đúng?

Vì đề yêu cầu:

```text id="7xgf9y"
multiple private media files
```

và:

```text id="u9gq5m"
không đổi URL hiện tại
```

Signed Cookies hoạt động bằng cách:

```text id="z6e5j4"
browser gửi cookie
CloudFront kiểm tra cookie
nếu hợp lệ → cho access
```

URL không cần thay đổi.

---

# 6. Flow hoạt động

## User chưa login/member

```text id="5cjlwm"
User → CloudFront
       ↓
Access denied
```

---

## User là paying subscriber

Application trả:

```text id="gby57l"
Set-Cookie headers
```

Browser lưu signed cookies.

---

## Sau đó user access media files

```text id="7a6nvs"
Browser + Signed Cookies
        ↓
CloudFront xác minh
        ↓
ALLOW
```

---

# 7. Vì sao Signed URL sai?

Signed URL phù hợp khi:

# bảo vệ từng file riêng lẻ

Ví dụ:

```text id="u0xj3f"
movie.mp4
```

Bạn phải tạo URL kiểu:

```text id="h7lwym"
https://cdn.example.com/movie.mp4?signature=abc
```

---

# Vấn đề

Đề nói:

```text id="jlwmqo"
don't change current URLs
```

Signed URL bắt buộc modify URL.

Ngoài ra nếu có nhiều files:

```text id="w7oyw8"
mỗi file phải tạo signed URL riêng
```

=> không tối ưu.

---

# 8. Vì sao Match Viewer sai?

Match Viewer là:

# Origin Protocol Policy

Nó chỉ quyết định:

```text id="4gw52s"
CloudFront ↔ Origin dùng HTTP hay HTTPS
```

Không liên quan member access.

---

# 9. Vì sao Field-Level Encryption sai?

Field-Level Encryption dùng để:

# mã hóa dữ liệu nhạy cảm user submit lên server

Ví dụ:

* credit card
* SSN
* personal info

Không dùng để:

```text id="f6r1bs"
protect download access to private files
```

---

# 10. Phân biệt cực quan trọng

| Feature                | Dùng cho gì                 |
| ---------------------- | --------------------------- |
| Signed URL             | Access 1 file cụ thể        |
| Signed Cookies         | Access nhiều private files  |
| Field-Level Encryption | Encrypt sensitive form data |
| Match Viewer           | HTTP/HTTPS policy           |
