# 1. Dịch câu hỏi

Một nền tảng mua sắm online đang được host trên một Auto Scaling Group gồm các Amazon EC2 Spot Instances và sử dụng **Amazon Aurora PostgreSQL** làm database.

Yêu cầu là tối ưu workload database trong cluster bằng cách:

* chuyển **production traffic** đến các instance có cấu hình/capacity cao
* chuyển các **reporting queries** từ nhân viên nội bộ đến các instance có cấu hình/capacity thấp

Cấu hình nào phù hợp nhất cho application và Aurora database cluster để đạt yêu cầu này?

---

# 2. Đáp án đúng

✅ **Create a custom endpoint in Aurora based on the specified criteria for the production traffic and another custom endpoint to handle the reporting queries.**

Dịch:

> Tạo một custom endpoint trong Aurora cho production traffic dựa trên tiêu chí cụ thể, và tạo một custom endpoint khác để xử lý các reporting queries.

# 4. Aurora Endpoint là gì?

Aurora không bắt app connect trực tiếp từng DB instance.

Nó dùng:

# Endpoint

Endpoint giống như “địa chỉ kết nối database”.

App connect vào endpoint, Aurora sẽ route đến instance phù hợp.

---

# 5. Các loại endpoint trong Aurora

| Endpoint                           | Dùng để làm gì                                        |
| ---------------------------------- | ----------------------------------------------------- |
| Cluster endpoint / Writer endpoint | Connect tới primary instance để write                 |
| Reader endpoint                    | Load balance read queries qua các read replicas       |
| Instance endpoint                  | Connect trực tiếp tới một DB instance cụ thể          |
| Custom endpoint                    | Connect tới một nhóm DB instances theo tiêu chí riêng |

---

# 6. Vì sao Custom Endpoint đúng?

Vì đề yêu cầu chia traffic theo loại instance:

```text
Production traffic → high-capacity instances
Reporting queries → low-capacity instances
```

Custom endpoint cho phép tạo endpoint trỏ tới một subset DB instances.

Ví dụ:

```text
prod-endpoint
   ↓
High-capacity replicas

report-endpoint
   ↓
Low-capacity replicas
```

---

# 7. Flow đúng

```text
Production App
      ↓
Aurora Custom Endpoint A
      ↓
High-capacity DB instances
```

```text
Internal Staff Reporting
      ↓
Aurora Custom Endpoint B
      ↓
Low-capacity DB instances
```

=> workload được tách riêng.

=> production không bị ảnh hưởng bởi report queries.

---

# 8. Vì sao các đáp án khác sai?

## ❌ Dùng reader endpoint cho cả production traffic và reporting queries

Reader endpoint chỉ load balance read traffic qua tất cả replicas.

Vấn đề:

```text
Nó không biết instance nào mạnh/yếu để route theo yêu cầu
```

Nó có thể gửi reporting queries vào high-capacity instance hoặc production traffic vào low-capacity instance.

=> không đáp ứng yêu cầu route theo capacity.

---

## ❌ Dùng instance endpoint cho production và cluster endpoint cho reporting

Sai vì:

### Instance endpoint

Connect vào đúng 1 instance cụ thể.

Không phù hợp để scale/load balance production traffic.

### Cluster endpoint

Cluster endpoint là writer endpoint.

Nó trỏ tới primary DB instance.

Phù hợp cho write traffic, không phù hợp cho reporting queries.

Ngoài ra không route theo high/low capacity.

---

## ❌ Do nothing

Aurora không tự động biết:

```text
production → high capacity
reporting → low capacity
```

Mặc định Aurora không route workload theo capacity như vậy.

Muốn làm được phải tạo custom endpoint.

# 9. Ví dụ dễ hiểu

Giả sử Aurora cluster có:

```text
db-r6g.4xlarge  → mạnh
db-r6g.4xlarge  → mạnh
db-r6g.large    → yếu
db-r6g.large    → yếu
```

Ta tạo:

```text
prod-custom-endpoint
  → db-r6g.4xlarge instances
```

```text
report-custom-endpoint
  → db-r6g.large instances
```

Application config:

```text
Production app dùng prod-custom-endpoint
Internal report dùng report-custom-endpoint
```
