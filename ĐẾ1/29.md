# 1. Dịch câu hỏi

Một website bán xe được host trên Amazon EC2 và lưu danh sách xe trong Amazon Aurora database (managed bởi Amazon RDS).

Khi một chiếc xe được bán:

* dữ liệu của xe phải bị xóa khỏi current listings
* đồng thời dữ liệu đó phải được gửi tới một distributed processing system

Giải pháp nào đáp ứng yêu cầu?

---

# 2. Đáp án đúng

✅ **Use an Aurora MySQL native function to invoke an AWS Lambda function whenever a vehicle listing is deleted. Configure the Lambda function to send the data to an Amazon SQS queue for the distributed processing system to consume.**

Dịch:

> Sử dụng native function của Aurora MySQL để invoke AWS Lambda mỗi khi vehicle listing bị xóa. Lambda sẽ gửi dữ liệu vào Amazon SQS để distributed processing system xử lý.

---

# 3. Keyword quan trọng của đề

## Keyword 1

# "whenever a vehicle listing is deleted"

Đây là keyword QUAN TRỌNG NHẤT.

Đề cần trigger khi:

# dữ liệu trong table thay đổi

cụ thể là:

# DELETE operation

---

## Keyword 2

# "forwarded to a distributed processing system"

=> cần gửi event/data sang hệ thống xử lý khác.

Pattern phổ biến:

```text
DB change
   ↓
Queue/Event
   ↓
Distributed consumers
```

---

## Keyword 3

# "Aurora MySQL"

Aurora MySQL có:

# native Lambda integration

=> đây là clue mạnh.

---

# 4. Aurora MySQL Native Function là gì?

Aurora MySQL hỗ trợ native function:

* lambda_sync
* lambda_async

Cho phép:

# gọi trực tiếp Lambda từ database

---

# Ví dụ

Khi row bị DELETE:

```sql
CALL mysql.lambda_async(...)
```

---

# 5. Flow đúng

```text
Vehicle deleted khỏi Aurora MySQL
        ↓
Aurora native function trigger Lambda
        ↓
Lambda gửi message vào SQS
        ↓
Distributed processing system consume từ SQS
```

---

# 6. Vì sao đây là đáp án đúng?

Vì đề yêu cầu:

```text
trigger theo DATA CHANGE
```

không phải infrastructure event.

Aurora native Lambda function có thể:

* trigger realtime
* bám sát row-level change
* dễ tích hợp event-driven architecture

---

# 7. Vì sao dùng SQS?

Distributed processing system thường cần:

* decoupling
* buffering
* async processing
* scaling consumers

SQS phù hợp hoàn hảo.

---

# 8. Vì sao RDS Event Subscription sai?

Đây là phần rất dễ nhầm.

---

# RDS Event Subscription chỉ monitor:

# infrastructure/operational events

Ví dụ:

* DB reboot
* failover
* backup completed
* maintenance
* storage low

---

# Nó KHÔNG monitor:

```text
INSERT
UPDATE
DELETE
```

trong table.

---

# 9. Vì sao SNS/SQS fanout options sai?

Các option đó đều dựa trên:

# RDS Event Subscription

Mà RDS Event Subscription không bắt được:

```text
DELETE vehicle listing
```

=> sai từ gốc.

---

# 10. Phân biệt cực quan trọng

| Service/Feature               | Dùng cho gì                  |
| ----------------------------- | ---------------------------- |
| RDS Event Subscription        | DB operational events        |
| Aurora native Lambda function | Trigger theo DB data changes |
| SQS                           | Distributed async processing |
| SNS                           | Fanout pub/sub               |
| Lambda                        | Event processing             |

# 11. RDS Event Subscription KHÔNG phải CDC

Nhiều người nhầm:


Nó không phải:

# CDC (Change Data Capture)
