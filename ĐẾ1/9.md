# 1. Dịch câu hỏi

Một ứng dụng Docker đang chạy trên một Amazon ECS Cluster phía sau Load Balancer và đang sử dụng Amazon DynamoDB rất nhiều.

Ứng dụng yêu cầu cải thiện hiệu suất database bằng cách:

* phân phối workload đồng đều
* sử dụng provisioned throughput hiệu quả hơn

Giải pháp nào nên được triển khai cho DynamoDB table?

# 2. Keyword quan trọng của đề

# "improved database performance"

# HOT PARTITION

# 4. Partition Key là gì?

Partition Key quyết định:


---

# 5. High-cardinality là gì?

Cardinality = số lượng giá trị khác nhau.

---

# High-cardinality

Có RẤT NHIỀU giá trị khác nhau.

Ví dụ:

| Attribute | Cardinality |
| --------- | ----------- |
| user_id   | cao         |
| email     | cao         |
| order_id  | cao         |

---

# Low-cardinality

Chỉ có ÍT giá trị.

Ví dụ:

| Attribute | Cardinality |
| --------- | ----------- |
| gender    | thấp        |
| country   | thấp        |
| status    | thấp        |

---

# 6. Vì sao High-cardinality tốt?

Ví dụ dùng:

```text id="v17odq"
partition key = user_id
```

Data:

```text id="h6sh4e"
user_1
user_2
user_3
user_4
```

DynamoDB sẽ phân tán:

```text id="y1h1es"
user_1 → Partition A
user_2 → Partition B
user_3 → Partition C
user_4 → Partition D
```

=> request được chia đều.

=> workload cân bằng.

=> throughput được dùng hiệu quả.

=> performance tốt.

---

# 7. Vì sao Low-cardinality nguy hiểm?

Ví dụ dùng:

```text id="r7rzpy"
partition key = gender
```

Chỉ có:

```text id="2r3tya"
male
female
```

Kết quả:

```text id="prl1v4"
rất nhiều request dồn vào vài partition
```

=> xảy ra:

# HOT PARTITION

---

# 8. Hot Partition là gì?

Là tình trạng:

```text id="mxax0u"
1 partition nhận quá nhiều traffic
```

Hậu quả:

* throttling
* latency tăng
* chậm hệ thống
* dùng throughput không hiệu quả

---

# 9. Đáp án đúng

## ✅ Use partition keys with high-cardinality attributes

Vì:

* workload được spread đều
* tránh hot partition
* tối ưu throughput
* improve DynamoDB performance

---

# 10. Vì sao các đáp án khác sai?

---

## ❌ Use partition keys with low-cardinality attributes

Đây gần như là:

# nguyên nhân gây hot partition

Ví dụ:

```text id="5mkxw7"
status = ACTIVE
```

Hầu hết request dồn vào cùng partition.

=> performance giảm mạnh.

---

## ❌ Reduce the number of partition keys

Ít partition key hơn:

=> ít phân phối dữ liệu hơn.

=> traffic càng tập trung.

=> dễ nghẽn hơn.

---

## ❌ Avoid using a composite primary key

Composite primary key:

```text id="gmifp8"
partition key + sort key
```

Thực tế rất hữu ích.

Ví dụ:

```text id="6e2w8p"
PK = user_id
SK = order_date
```

Giúp:

* query hiệu quả
* organize data tốt hơn
* scaling tốt hơn

=> không nên tránh.

---

# 11. Ví dụ thực tế cực dễ hiểu

---

# ❌ Bad Design

```text id="3ahjj9"
PK = country
```

Data:

```text id="dw2dyo"
US
US
US
US
VN
JP
```

90% traffic vào US.

=> partition US quá tải.

---

# ✅ Good Design

```text id="oz7vhs"
PK = user_id
```

Ví dụ:

```text id="uy8jv0"
100001
100002
100003
100004
```

=> request được trải đều toàn bộ partitions.

=> scale rất tốt.

# Mapping nhanh

| Requirement               | Giải pháp               |
| ------------------------- | ----------------------- |
| Even workload             | High-cardinality PK     |
| Avoid hot partition       | Spread partition keys   |
| Efficient throughput      | Many distinct PK values |
| Better query organization | Composite primary key   |
