# Dịch đề

Một bộ web applications được host trên:

```text
Auto Scaling Group của EC2
```

trải trên 3 Availability Zones.

Hệ thống dùng default settings.

Có Application Load Balancer chuyển request tới target group tương ứng theo URL path.

Do traffic thấp, scale-in policy được kích hoạt.

Hỏi:

# EC2 instance nào sẽ bị terminate đầu tiên?

---

# Đáp án đúng

✅

```text
The EC2 instance launched from the oldest launch template.
```

Dịch:

> EC2 instance được launch từ launch template cũ nhất sẽ bị terminate trước.

---

# Phân tích keyword

## Keyword 1

```text
Auto Scaling group
```

=> liên quan EC2 Auto Scaling.

---

## Keyword 2

```text
default settings
```

=> dùng default termination policy.

---

## Keyword 3

```text
scale-in policy has been triggered
```

=> Auto Scaling cần chọn instance để terminate.

---

# Default termination policy hoạt động thế nào?

Khi scale-in, Auto Scaling không chọn bừa ngay.

Nó có thứ tự ưu tiên.

---

# Bước 1: Chọn Availability Zone có nhiều instance nhất

Mục tiêu là giữ các AZ cân bằng.

Ví dụ:

```text
AZ-A: 3 instances
AZ-B: 2 instances
AZ-C: 2 instances
```

=> chọn AZ-A trước.

---

# Bước 2: Trong AZ đó, chọn instance dùng launch template cũ nhất

Ví dụ trong AZ-A:

```text
Instance 1 → launch template version 1
Instance 2 → launch template version 2
Instance 3 → launch template version 2
```

=> terminate instance dùng version 1 trước.

---

# Bước 3: Nếu vẫn còn nhiều instance giống nhau

Auto Scaling chọn instance gần tới billing hour tiếp theo nhất.

Mục đích là tối ưu chi phí.

---

# Bước 4: Nếu vẫn hòa

Mới chọn random.

---

# Vì sao đáp án đúng là oldest launch template?

Vì đề nói:

```text
default settings
```

Default termination policy ưu tiên:

```text
cân bằng AZ
→ launch template cũ nhất
→ gần billing hour
→ random
```

Trong các đáp án, đáp án khớp nhất là:

```text
instance launched from the oldest launch template
```

---

# Vì sao các đáp án khác sai?

## ❌ Instance chạy lâu nhất

Sai.

Auto Scaling không ưu tiên terminate instance chạy lâu nhất theo thời gian uptime.

---

## ❌ Instance có ít user sessions nhất

Sai.

Auto Scaling không biết user session trong app của bạn.

Nó không chọn dựa trên session.

Muốn bảo vệ session thì app cần dùng:

* sticky sessions
* external session store
* ElastiCache
* DynamoDB

---

## ❌ Random

Sai một phần.

Random chỉ xảy ra ở bước cuối nếu các tiêu chí trước đó hòa nhau.

Không phải mặc định chọn random ngay.

---

# Bảng nhớ nhanh

| Tiêu chí                      | Có dùng trong default termination? |
| ----------------------------- | ---------------------------------- |
| AZ có nhiều instance nhất     | ✅                                  |
| Oldest launch template/config | ✅                                  |
| Closest to next billing hour  | ✅                                  |
| Random                        | ✅ cuối cùng                        |
| Longest running time          | ❌                                  |
| Least user sessions           | ❌                                  |

# Chốt

EC2 bị terminate đầu tiên là instance nằm trong AZ cần giảm tải và được launch từ:

```text
oldest launch template
```
Mình sẽ mô tả lại toàn bộ bối cảnh thực tế của câu này để bạn hiểu Auto Scaling đang làm gì.
