# 1. Dịch câu hỏi

Một công ty liên tục gặp sự cố mất dịch vụ tại **Availability Zone** nơi Amazon RDS database instance đang được triển khai.

Mỗi lần sự cố xảy ra, công ty bị:

# mất hoàn toàn quyền truy cập vào database

Giải pháp nào nên được triển khai để tránh mất quyền truy cập database nếu sự cố này xảy ra lần nữa?

---

# 2. Đáp án đúng

✅ **Enable Multi-AZ failover**

Dịch:

> Bật Multi-AZ failover cho Amazon RDS.

---

# 3. Keyword quan trọng của đề

## Keyword 1

# "repeated outages in the Availability Zone"

Sự cố xảy ra ở cấp:

# Availability Zone

Không phải chỉ lỗi CPU/RAM hay thiếu tài nguyên.

---

## Keyword 2

# "complete loss of access to the database"

Database hiện tại có thể đang chạy ở 1 AZ.

Khi AZ đó lỗi:

```text
DB không truy cập được
```

---

## Keyword 3

# "prevent losing database access"

Đề cần:

# High Availability

Không phải backup đơn thuần.

---

# 4. Multi-AZ RDS là gì?

Multi-AZ là cơ chế triển khai RDS có:

```text
Primary DB instance ở AZ A
Standby DB instance ở AZ B
```

Dữ liệu được replicate đồng bộ.

---

# 5. Flow hoạt động

## Bình thường

```text
AZ A
Primary RDS
   ↓ synchronous replication
AZ B
Standby RDS
```

---

## Khi AZ A lỗi

```text
AZ A lỗi
   ↓
RDS tự động failover
   ↓
Standby ở AZ B trở thành Primary
```

=> app vẫn có thể truy cập database sau failover.

---

# 6. Vì sao Multi-AZ đúng?

Vì nó giải quyết đúng vấn đề:

```text
AZ-level outage
```

Multi-AZ cung cấp:

* standby instance ở AZ khác
* synchronous replication
* automatic failover
* high availability
* giảm downtime

---

# 7. Vì sao snapshot sai?

Snapshot dùng cho:

# backup / restore

Không dùng cho high availability realtime.

Nếu AZ lỗi, bạn phải:

```text
restore snapshot → tạo DB mới → đổi endpoint/config
```

=> mất thời gian, downtime lâu.

---

# 8. Vì sao read replica sai?

Read replica dùng chính cho:

# scale read traffic

Nó không tự động failover như Multi-AZ.

Nếu primary lỗi:

```text
phải promote read replica thủ công
```

=> không đáp ứng yêu cầu tránh mất access khi sự cố xảy ra.

---

# 9. Vì sao tăng DB instance size sai?

Increase instance size giúp:

* thêm CPU
* thêm RAM
* cải thiện performance

Nhưng không giải quyết:

```text
AZ bị outage
```

DB vẫn nằm trong cùng AZ đó.

=> vẫn mất access nếu AZ lỗi.

---

# 10. Multi-AZ vs Read Replica

| Tiêu chí        | Multi-AZ                   | Read Replica               |
| --------------- | -------------------------- | -------------------------- |
| Mục đích chính  | High Availability          | Read scaling               |
| Replication     | Synchronous                | Asynchronous               |
| Failover        | Automatic                  | Không tự động như Multi-AZ |
| Dùng để đọc?    | Standby không phục vụ read | Có thể phục vụ read        |
| Xử lý AZ outage | ✅ Tốt                      | ❌ Không phải chính         |

---

# 11. Cách nhớ exam

Nếu đề có:

```text
RDS
Availability Zone outage
database unavailable
high availability
automatic failover
```

=> chọn:

# RDS Multi-AZ

---

Nếu đề có:

```text
read-heavy workload
offload read traffic
reporting queries
```

=> chọn:

# Read Replica

---

# 12. Chốt ngắn gọn

Vấn đề là:

```text
DB mất access khi AZ lỗi
```

Giải pháp đúng:

```text
Enable RDS Multi-AZ
   ↓
Có standby DB ở AZ khác
   ↓
Tự động failover khi primary/AZ lỗi
```
