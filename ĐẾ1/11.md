# 1. Dịch câu hỏi

Một hệ thống CMS (Content Management System) đang được host trên một fleet gồm nhiều Amazon EC2 On-Demand instances có Auto Scaling và sử dụng Amazon Aurora làm database.

Hiện tại, các file tài liệu mà người dùng upload lên đang được lưu trong một Amazon EBS volume gắn vào EC2.

Hiệu năng hệ thống đang chậm và manager yêu cầu team cải thiện kiến trúc.

Trong trường hợp này, giải pháp nào nên được triển khai để đạt được:

* scalable
* highly available
* POSIX-compliant shared file system

---

# 2. Đáp án đúng

✅ Use Amazon EFS to provide a shared file system for concurrent access to data

---

# 3. Keyword quan trọng của đề

---

## Keyword 1

# "fleet of auto-scaled EC2 instances"

Đây là keyword QUAN TRỌNG.

Nghĩa là:

```text
nhiều EC2 instances cùng chạy
```

và còn auto scaling.

=> storage cần:

# shared access

---

## Keyword 2

# "users upload file documents"

=> cần:

# file storage

Không phải block storage.

Không phải object storage đơn thuần.

---

## Keyword 3

# "performance has been observed to be slow"

=> kiến trúc hiện tại có vấn đề.

---

## Keyword 4

# "POSIX-compliant shared file system"

Đây là keyword MẠNH NHẤT.

POSIX-compliant file system nghĩa là:

* mount như Linux filesystem
* hỗ trợ file locking
* hỗ trợ concurrent access
* dùng NFS semantics

=> keyword gần như chỉ thẳng tới:

# Amazon EFS

---

# 4. Kiến trúc hiện tại có vấn đề gì?

Hiện tại:

```text
EC2
  ↓
Attached EBS volume
```

---

# Vấn đề của EBS

EBS là:

# block storage

Nó phù hợp cho:

* OS disk
* database disk
* single instance storage

---

# Nhưng hệ thống này có nhiều EC2

```text
EC2-1
EC2-2
EC2-3
```

Nếu mỗi instance có EBS riêng:

=> file không đồng bộ.

---

# Nếu attach chung EBS?

EBS Multi-Attach có nhiều hạn chế:

* chỉ trong cùng AZ
* không phù hợp CMS shared filesystem
* không scale tốt như EFS

---

# 5. Vì sao EFS đúng?

Amazon EFS là:

# managed shared file system

Nó cho phép:

```text
EC2-1
EC2-2
EC2-3
    ↓
   EFS
```

Tất cả EC2 cùng đọc/ghi 1 filesystem.

---

# EFS hỗ trợ gì?

## ✅ Shared concurrent access

Nhiều EC2 truy cập cùng lúc.

---

## ✅ POSIX-compliant

Hoạt động như filesystem Linux bình thường.

---

## ✅ Highly Available

EFS multi-AZ.

---

## ✅ Scalable

Tự động scale dung lượng.

---

# 6. Vì sao S3 sai?

S3 là:

# object storage

Không phải filesystem.

---

# CMS thường cần:

* file locking
* mount filesystem
* POSIX semantics

S3 không hỗ trợ các thứ này.

---

# So sánh dễ hiểu

## S3

Giống:

```text
kho object
```

Upload/download object.

---

## EFS

Giống:

```text
shared network drive
```

Có thể:

```bash
cd
ls
mkdir
chmod
```

như filesystem Linux thật.

---

# 7. Vì sao nâng cấp EBS sai?

Provisioned IOPS SSD:

```text
gp3 / io1 / io2
```

chỉ tăng performance.

---

Nhưng vấn đề chính KHÔNG phải chỉ là tốc độ disk.

Mà là:

# shared filesystem cho nhiều EC2

EBS không giải quyết tốt bài toán này.

---

# 8. Vì sao ElastiCache sai?

ElastiCache là:

# in-memory cache

Dùng cho:

* cache session
* cache query
* giảm DB latency

Không phải:

# file storage

---

# 9. Phân biệt cực quan trọng

| Service | Loại storage   |
| ------- | -------------- |
| EBS     | Block storage  |
| EFS     | File storage   |
| S3      | Object storage |

---

# 10. Khi nào dùng gì?

---

# EBS

Dùng cho:

* 1 EC2
* OS disk
* database

---

# EFS

Dùng cho:

* nhiều EC2 cùng access
* shared filesystem
* Linux filesystem
* POSIX

---

# S3

Dùng cho:

* object storage
* image/video/archive
* static assets

---
