# Dịch đề

Một công ty đang migrate application lên AWS.

Một trong các hệ thống của họ cần database có khả năng:

* scale globally
* handle frequent schema changes

Ứng dụng phải:

* không downtime
* không performance issue
* mỗi khi schema thay đổi

Ngoài ra database còn phải:

* phản hồi low latency
* xử lý high-traffic queries

Database nào phù hợp nhất?

---

✅

```text
Amazon DynamoDB
```

# Schema là gì?

Schema là:

# cấu trúc dữ liệu

Ví dụ relational DB:

```text
Users Table
- id INT
- name VARCHAR
- age INT
```

Nếu muốn thêm field:

```text
address
```

=> phải ALTER TABLE.

---

# Vấn đề của relational database

RDS/Aurora là relational database.

Nó có:

# rigid schema

Tức là schema cứng.

---

# Khi schema thay đổi thường xuyên

Ví dụ:

```text
thêm column
đổi datatype
thêm relationship
```

=> relational DB có thể:

* downtime
* lock table
* giảm performance

---

# DynamoDB khác gì?

DynamoDB là:

# NoSQL database

Schema rất flexible.

---

# Ví dụ DynamoDB

Item 1:

```json
{
  "userId": 1,
  "name": "John"
}
```

Item 2:

```json
{
  "userId": 2,
  "name": "Alice",
  "address": "Tokyo",
  "vip": true
}
```

Không cần ALTER TABLE.

Không cần migration schema phức tạp.

---

# Vì sao DynamoDB phù hợp?

---

# ✅ Flexible schema

Thêm field mới rất dễ.

Không downtime.

---

# ✅ Scale globally

DynamoDB scale cực mạnh.

Có thể dùng:

# Global Tables

cho multi-region.

---

# ✅ Low latency

DynamoDB cho:

```text
single-digit millisecond latency
```

rất phù hợp high traffic.

---

# ✅ High traffic queries

DynamoDB được thiết kế cho:

* massive scale
* millions requests
* low latency

---

# Vì sao các đáp án khác sai?

---

# ❌ Amazon RDS Multi-AZ

RDS là:

# relational database

Schema cứng.

Multi-AZ chỉ giúp:

# high availability

Không giải quyết schema flexibility.

---

# ❌ Aurora + Read Replicas

Aurora vẫn là:

# relational database

Read Replica chỉ tăng read scaling.

Không xử lý tốt schema thay đổi thường xuyên.

---

# ❌ Redshift

Redshift dùng cho:

# OLAP / Data Warehouse

Ví dụ:

* analytics
* BI reporting
* big query aggregation

Không phù hợp transactional application.

---

# So sánh cực hay thi

| Database | Type           | Schema    |
| -------- | -------------- | --------- |
| RDS      | Relational     | rigid     |
| Aurora   | Relational     | rigid     |
| DynamoDB | NoSQL          | flexible  |
| Redshift | Data Warehouse | analytics |

# Chốt kiến thức cực quan trọng

## Relational DB

Ưu điểm:

* joins mạnh
* ACID
* consistency tốt

Nhược điểm:

* schema rigid
* scale khó hơn

---

## DynamoDB

Ưu điểm:

* flexible schema
* scale cực mạnh
* low latency
* serverless

Nhược điểm:

* không mạnh về joins/complex relations
