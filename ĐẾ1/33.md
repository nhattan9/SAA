# 1. Dịch câu hỏi

Một công ty công nghệ đã thực hiện phân tích:

# TCO (Total Cost of Ownership)

để so sánh giữa:

* dùng Amazon S3
* mua thêm storage hardware

Kết quả là:

```text
1200 nhân viên
```

sẽ được cấp quyền dùng Amazon S3 để lưu tài liệu cá nhân.

Yêu cầu:

* hỗ trợ Single Sign-On (SSO)
* tích hợp với corporate AD/LDAP directory
* mỗi user chỉ được truy cập folder riêng của mình trong S3 bucket

Cần chọn HAI giải pháp phù hợp.

---

# 2. Đáp án đúng

✅ **Set up a Federation proxy or an Identity provider, and use AWS STS to generate temporary tokens**

✅ **Configure an IAM role and an IAM Policy to access the bucket**

---

# 3. Keyword quan trọng

## Keyword 1

# "Single Sign-On"

=> người dùng login bằng account công ty.

Không muốn tạo account AWS riêng cho từng user.

=> nghĩ tới:

# Federation

---

## Keyword 2

# "corporate AD or LDAP"

=> dùng identity từ hệ thống công ty.

Ví dụ:

* Microsoft Active Directory
* LDAP
* Okta
* ADFS

---

## Keyword 3

# "1200 employees"

Đây là keyword rất quan trọng.

=> KHÔNG nên tạo:

```text
1200 IAM users
```

---

## Keyword 4

# "restrict access for each individual user"

=> cần:

# IAM policy / role-based access control

để mỗi user chỉ thấy folder của mình.

---

# 4. Giải pháp đúng hoạt động thế nào?

## Flow tổng thể

```text
Corporate AD / LDAP
        ↓
Identity Provider / Federation
        ↓
AWS STS
        ↓
Temporary Credentials
        ↓
IAM Role + IAM Policy
        ↓
Access đúng folder trong S3
```

---

# 5. AWS STS là gì?

STS =

# Security Token Service

Nó cấp:

# temporary credentials

cho user federated.

---

# Ví dụ

User login bằng corporate AD:

```text
john@company.com
```

---

Sau khi authenticate:

```text
AWS STS cấp temporary token
```

User dùng token đó access S3.

---

# 6. Vì sao IAM Role + Policy cần thiết?

SSO chỉ giải quyết:

# authentication

(tức là user là ai)

---

IAM Role + Policy giải quyết:

# authorization

(tức là user được làm gì)

---

# Ví dụ policy

User John chỉ được access:

```text
s3://company-bucket/john/*
```

Không được xem:

```text
s3://company-bucket/mary/*
```

---

# 7. Vì sao không tạo 1200 IAM Users?

Sai vì:

* cực khó quản lý
* duplicate account management
* không dùng SSO thật sự
* không integrate AD/LDAP tốt
* không scalable

AWS best practice:

# Federation + STS

---

# 8. Vì sao WorkDocs sai?

Amazon WorkDocs là:

# collaboration/content management service

Không phải cơ chế chuẩn để:

```text
AD/LDAP federation + S3 folder authorization
```

---

# Ví dụ

# 11. Phân biệt nhanh

| Thành phần       | Vai trò                        |
| ---------------- | ------------------------------ |
| AD / LDAP        | Identity source                |
| Federation / IdP | Authenticate user              |
| AWS STS          | Generate temporary credentials |
| IAM Role         | Quyền truy cập AWS             |
| IAM Policy       | Restrict access cụ thể         |
| S3 Bucket Policy | Bucket-level permissions       |
