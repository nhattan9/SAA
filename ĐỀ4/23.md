## Câu hỏi 23

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company has an OLTP application that is hosted in an Amazon ECS cluster using the Fargate launch type. It has an Amazon RDS database that stores data of its production website. The Data Analytics team needs to run queries against the database to track and audit all user transactions. These query operations against the production database must not impact application performance in any way.
>
> Which of the following is the MOST suitable and cost-effective solution that you should implement?

---

### 🔤 DỊCH CHI TIẾT

Công ty có ứng dụng **OLTP** (xử lý giao dịch trực tuyến) chạy trên ECS Fargate, dùng **Amazon RDS** làm database. Data Analytics team cần chạy query để theo dõi và audit tất cả giao dịch người dùng. Yêu cầu: các query này **không được ảnh hưởng đến hiệu năng production**.

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Tách biệt read workload** khỏi production database — analytics queries không được chạy trực tiếp trên primary instance.
2. **Cost-effective** — không cần giải pháp đắt tiền hay phức tạp.
3. Từ khóa: **"read queries"** từ analytics team → phù hợp với **Read Replica**.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: RDS Read Replica — cho Data Analytics team query từ replica**

- Read Replica là bản sao chỉ đọc của production database, được sync liên tục qua **asynchronous replication**. ✅
- Analytics team query trên Read Replica → production primary instance **không bị ảnh hưởng**. ✅
- Cost-effective: chỉ tốn thêm chi phí của một RDS instance — đơn giản, không cần migration hay setup phức tạp. ✅
- Có thể tạo tối đa **5 Read Replicas** cho một RDS instance, có thể đặt ở region khác. ✅

---

❌ **Đáp án sai 1: Multi-AZ — query từ standby instance**

- Với Multi-AZ, **standby instance không thể query được** — nó chỉ tồn tại để **automatic failover**, không phải để phục vụ read traffic.
- Đây là sự nhầm lẫn kinh điển giữa Multi-AZ (HA) và Read Replica (read scaling).

---

❌ **Đáp án sai 2: Amazon Redshift — migrate production database vào Redshift**

- **Redshift là OLAP** (Online Analytical Processing) — thiết kế cho analytics phức tạp trên data warehouse, không phải OLTP.
- Migrate toàn bộ production database sang Redshift là tốn kém, phức tạp và không phù hợp với use case này.
- Overkill so với việc chỉ đơn giản tạo một Read Replica.

---

❌ **Đáp án sai 3: Upgrade RDS instance lên instance type lớn hơn**

- Upgrade instance type tốn chi phí cao hơn — không cost-effective.
- Quan trọng hơn: dù upgrade instance lớn hơn, analytics queries vẫn chạy trên **cùng production database** → vẫn có thể ảnh hưởng đến application performance.
- Không giải quyết được vấn đề cốt lõi: tách biệt workload.

---

### 📊 KẾT LUẬN & BÀI HỌC

**So sánh Multi-AZ vs Read Replica — hay nhầm nhất trong đề thi:**

| Tiêu chí | Multi-AZ | Read Replica |
|---|---|---|
| Mục đích | High Availability, automatic failover | Scale read traffic |
| Replication | Synchronous | Asynchronous |
| Có thể query không? | ❌ Standby không query được | ✅ Có thể query read-only |
| Dùng khi | Cần failover tự động | Cần giảm tải read cho production |
| Số lượng | 1 standby | Tối đa 5 replicas |

**OLTP vs OLAP — phân biệt nhanh:**

| | OLTP | OLAP |
|---|---|---|
| Là gì | Xử lý giao dịch real-time | Phân tích dữ liệu lớn |
| Ví dụ | Đặt hàng, thanh toán, đăng nhập | Báo cáo doanh thu, data mining |
| AWS service | RDS, Aurora, DynamoDB | Redshift, Athena, EMR |
| Query kiểu | Nhiều query nhỏ, nhanh | Ít query lớn, phức tạp |

💡 **Công thức ghi nhớ:**

> **"Query không ảnh hưởng production"** → **Read Replica** (tách read workload)

> **"Multi-AZ standby"** = không query được — chỉ dùng cho failover

> **Redshift** = OLAP/data warehouse, không phải giải pháp cho OLTP read scaling

> **Upgrade instance** = tốn tiền + vẫn không tách được workload → không phải giải pháp đúng
