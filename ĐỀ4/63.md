## Câu hỏi 63

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> An Auto Scaling group of Amazon EC2 Linux instances has basic monitoring enabled in Amazon CloudWatch. The Solutions Architect noticed that the ASG is not launching more instances as it should be, even though the servers already have high memory usage.
>
> Which of the following options should the Architect implement to solve this issue?

---

### 🔤 DỊCH CHI TIẾT

ASG không tự động scale out dù **memory usage cao**. Nguyên nhân: CloudWatch **mặc định không theo dõi memory usage** → ASG không có metric để trigger scale out.

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

**Vấn đề gốc rễ**: CloudWatch mặc định chỉ theo dõi:
- CPU Utilization ✅
- Network In/Out ✅
- Disk Read/Write ✅
- **Memory Usage ❌ — không có sẵn**

→ Phải cài **CloudWatch Agent** để thu thập memory metric → dùng metric đó trigger ASG.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: CloudWatch Unified Agent + SSM Parameter Store + Custom Metric + ASG scaling**

- **CloudWatch Unified Agent** thu thập memory usage từ EC2 — metric không có sẵn trong CloudWatch. ✅
- **SSM Parameter Store** lưu CloudWatch agent configuration — quản lý tập trung cho tất cả instances trong ASG. ✅
- Tạo **aggregated custom metric** về memory → dùng làm trigger cho ASG scale out. ✅
- Giải quyết đúng gốc rễ vấn đề. ✅

---

❌ **Đáp án sai 1: Enable detailed monitoring + custom metrics**

- **Detailed monitoring** chỉ tăng tần suất collect **standard metrics** (từ 5 phút → 1 phút) — không thêm memory metric.
- Memory vẫn không được theo dõi dù bật detailed monitoring.
- Setup phức tạp hơn so với cài CloudWatch Agent.

---

❌ **Đáp án sai 2: Amazon Comprehend + SageMaker**

- **Amazon Comprehend** là **NLP service** — phân tích văn bản, không liên quan đến server metrics.
- **SageMaker** là ML platform — không phải monitoring tool.
- Hoàn toàn sai mục đích.

---

❌ **Đáp án sai 3: Amazon Rekognition + AWS Well-Architected Tool**

- **Rekognition** là **image recognition** — nhận dạng ảnh, khuôn mặt, không liên quan memory monitoring.
- **Well-Architected Tool** là framework đánh giá kiến trúc — không trigger scale events.
- Hoàn toàn sai mục đích.

---

### 📊 KẾT LUẬN & BÀI HỌC

**CloudWatch default metrics vs custom metrics:**

| Metric | Có sẵn trong CloudWatch? | Cách lấy |
|---|---|---|
| CPU Utilization | ✅ | Mặc định |
| Network In/Out | ✅ | Mặc định |
| Disk Read/Write | ✅ | Mặc định |
| **Memory Usage** | ❌ | **Cần CloudWatch Agent** |
| **Disk Space Used** | ❌ | **Cần CloudWatch Agent** |
| Custom app metrics | ❌ | CloudWatch Agent hoặc SDK |

**Vai trò của SSM Parameter Store trong bài toán này:**

```
SSM Parameter Store lưu CloudWatch Agent config
    │
    ↓
Tất cả EC2 instances trong ASG đọc config từ đây
    │ (không cần cấu hình từng instance)
    ↓
CloudWatch Agent thu thập memory metrics
    │
    ↓
CloudWatch Custom Metric: MemoryUsage%
    │
    ↓
ASG Scaling Policy trigger khi MemoryUsage > 80%
```

💡 **Công thức ghi nhớ:**

> **"Memory/Disk metric + ASG không scale"** → CloudWatch **không có sẵn memory metric** → cài **CloudWatch Unified Agent**

> **SSM Parameter Store** = lưu CloudWatch Agent config tập trung → áp dụng cho nhiều instance cùng lúc

> **Detailed monitoring** = tăng tần suất standard metrics, **không thêm** memory metric

> **Comprehend** = NLP | **Rekognition** = image | **SageMaker** = ML → không liên quan monitoring
