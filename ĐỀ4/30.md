## Câu hỏi 30

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A solutions architect is tasked with designing a scalable infrastructure solution for a business that runs Amazon EKS to execute container applications. Since the company's workload varies throughout the day, they want to make sure that its underlying infrastructure automatically scales in and out in response to demand.
>
> Which of the following would meet the requirements with the LEAST amount of operational overhead?

---

### 🔤 DỊCH CHI TIẾT

Công ty dùng **Amazon EKS** để chạy container applications. Workload biến động trong ngày, cần hạ tầng **tự động scale in/out** theo demand với **ít operational overhead nhất**.

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **EKS auto-scaling** — cần scale cả ở tầng Pod lẫn tầng Node.
2. **Least operational overhead** — ưu tiên giải pháp native, tích hợp sẵn với Kubernetes, không cần cấu hình phức tạp.
3. Phân biệt **Pod scaling** (Horizontal Pod Autoscaler) vs **Node scaling** (Cluster Autoscaler).

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: Kubernetes Metrics + Kubernetes Cluster Autoscaler**

- **Kubernetes Metrics Server** cung cấp dữ liệu về CPU/memory usage của các Pods — là nguồn dữ liệu cho **Horizontal Pod Autoscaler (HPA)**. ✅
- **HPA** tự động scale số lượng Pods dựa trên CPU utilization — không cần cài thêm gì, là standard API resource trong Kubernetes. ✅
- **Cluster Autoscaler** tự động điều chỉnh số lượng **nodes** trong cluster khi Pods không thể schedule do thiếu tài nguyên. ✅
- Cả hai là giải pháp **native Kubernetes**, tích hợp tốt với EKS, operational overhead thấp nhất. ✅

---

❌ **Đáp án sai 1: Edge-optimized API Gateway + EKS**

- **Edge-optimized API endpoint** dùng để route API requests qua CloudFront đến người dùng phân tán địa lý — không liên quan đến scaling.
- Không giải quyết yêu cầu auto-scaling infrastructure.

---

❌ **Đáp án sai 2: EC2 Auto Scaling Groups với custom scaling policies**

- ASG có thể scale EKS worker nodes, nhưng cần **cấu hình và quản lý scaling policies thủ công** — operational overhead cao hơn.
- Không có sự tích hợp native với Kubernetes workload như Cluster Autoscaler.
- Cluster Autoscaler thực chất cũng dùng ASG ở backend, nhưng tự động hóa hoàn toàn — không cần tự viết custom policies.

---

❌ **Đáp án sai 3: CloudWatch Alarms theo dõi CPU/request count**

- CloudWatch Alarms chỉ **monitor và alert** — không tự động scale EKS cluster.
- Muốn dùng CloudWatch để trigger scaling thì phải kết hợp thêm nhiều thứ (Lambda, ASG policies...) — tăng operational overhead.
- Không phải giải pháp native cho EKS scaling.

---

### 📊 KẾT LUẬN & BÀI HỌC

**Hai tầng Auto-scaling trong EKS:**

```
Tầng Pod (ứng dụng):
  Kubernetes Metrics Server → cung cấp metrics
  Horizontal Pod Autoscaler (HPA) → scale số lượng Pods
                    ↓ (khi node hết tài nguyên)
Tầng Node (hạ tầng):
  Cluster Autoscaler / Karpenter → scale số lượng EC2 nodes
```

**So sánh hai EKS Autoscaling solutions:**

| | Cluster Autoscaler | Karpenter |
|---|---|---|
| Cơ chế | Dùng ASG để thêm/bớt nodes | Provision EC2 trực tiếp, không qua ASG |
| Tốc độ scale | Chậm hơn | Nhanh hơn |
| Linh hoạt instance type | Thấp hơn | Cao hơn (chọn instance type tối ưu) |
| Phù hợp | Môi trường đã có ASG | Môi trường mới, cần tối ưu chi phí |

**Kubernetes Autoscaling components:**

| Component | Vai trò |
|---|---|
| **Metrics Server** | Thu thập CPU/memory metrics từ Pods |
| **HPA** (Horizontal Pod Autoscaler) | Scale số lượng Pods theo metrics |
| **VPA** (Vertical Pod Autoscaler) | Điều chỉnh resource requests của Pod |
| **Cluster Autoscaler** | Scale số lượng Nodes trong cluster |
| **Karpenter** | Provision nodes nhanh hơn, thay thế Cluster Autoscaler |

💡 **Công thức ghi nhớ:**

> **EKS auto-scaling với least overhead** → **Kubernetes Metrics + Cluster Autoscaler** (native Kubernetes solution)

> **Pod scaling** → HPA + Metrics Server | **Node scaling** → Cluster Autoscaler hoặc Karpenter

> **EC2 ASG với custom policies** = overhead cao hơn Cluster Autoscaler — không phải giải pháp tối ưu cho EKS

> **CloudWatch Alarms** = chỉ monitor, không tự scale — cần thêm nhiều component mới hoạt động
