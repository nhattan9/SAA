## Câu hỏi 60

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company needs to implement a solution that will process real-time streaming data of its users across the globe. This will enable them to track and analyze globally-distributed user activity on their website and mobile applications, including clickstream analysis. The solution should process the data in close geographical proximity to their users and respond to user requests at low latencies.

---

### 🔤 DỊCH CHI TIẾT

Cần xử lý **real-time streaming data** từ người dùng toàn cầu (clickstream analysis). Yêu cầu: xử lý **gần người dùng về mặt địa lý** và **low latency**.

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Gần người dùng địa lý + low latency** → cần xử lý tại **Edge Locations** → **Lambda@Edge**.
2. **Real-time streaming data** → **Kinesis** (không phải Athena).
3. **Lưu kết quả bền vững** → S3.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: CloudFront + Lambda@Edge + Kinesis + S3**

- **Lambda@Edge** chạy code tại **CloudFront Edge Locations** gần người dùng nhất — xử lý request ngay tại edge, không cần về origin. ✅
- **Kinesis Data Streams** xử lý **real-time streaming data** (clickstream, user activity). ✅
- **S3** lưu kết quả bền vững. ✅
- Không cần provision server — serverless, globally distributed. ✅

---

❌ **Đáp án sai 1: CloudFront + Lambda@Edge + Amazon Athena**

- Lambda@Edge đúng, nhưng **Amazon Athena** là **interactive SQL query service** — dùng để query data đã có trong S3.
- Athena **không xử lý real-time streaming data** — không phù hợp thay thế Kinesis.

---

❌ **Đáp án sai 2: CloudFront + Route 53 Geoproximity + Kinesis**

- **Route 53** chỉ **route DNS traffic** — không có khả năng **compute/xử lý data**.
- Route 53 chỉ quyết định request đi đến server nào, không xử lý data gần người dùng.
- Thiếu Lambda@Edge → không có edge computing capability.

---

❌ **Đáp án sai 3: CloudFront + Route 53 Latency-based + Kinesis**

- Tương tự đáp án sai 2: Route 53 chỉ routing, không processing.
- Latency-based routing chọn region có latency thấp nhất, nhưng vẫn phải về AWS region — không phải Edge Location.

---

### 📊 KẾT LUẬN & BÀI HỌC

**Lambda@Edge vs Route 53 — phân biệt quan trọng:**

| | Lambda@Edge | Route 53 |
|---|---|---|
| Chạy ở đâu | CloudFront **Edge Locations** | DNS layer |
| Có thể process data | ✅ Có (compute) | ❌ Không (chỉ routing) |
| Gần người dùng | ✅ Cực gần (Edge) | Tương đối |
| Dùng cho | Edge computing, personalization, auth | DNS routing |

**Pipeline đúng cho bài toán:**

```
User (global)
    │ request
    ↓
CloudFront Edge Location (gần user nhất)
    │ Lambda@Edge xử lý tại đây
    ↓
Kinesis Data Streams (real-time streaming)
    │ xử lý clickstream data
    ↓
Amazon S3 (lưu kết quả)
```

**Phân biệt Kinesis vs Athena:**

| | Kinesis | Amazon Athena |
|---|---|---|
| Xử lý | **Real-time streaming** | Query data đã có trong S3 |
| Dùng cho | Clickstream, IoT, real-time analytics | Ad-hoc SQL query trên S3 |
| Real-time | ✅ | ❌ (batch query) |

💡 **Công thức ghi nhớ:**

> **"Xử lý gần người dùng địa lý + low latency + edge computing"** → **Lambda@Edge**

> **"Real-time streaming data"** → **Kinesis** (không phải Athena)

> **Route 53** = routing only, **không có compute** → không xử lý được data

> **Athena** = SQL query trên S3 data có sẵn, không phải real-time stream processor
