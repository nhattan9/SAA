## Câu hỏi 35

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company must integrate the Lightweight Directory Access Protocol (LDAP) directory service from the on-premises data center to the AWS VPC using IAM. The identity store which is currently being used is not compatible with SAML.
>
> Which of the following provides the most valid approach to implement the integration?

---

### 🔤 DỊCH CHI TIẾT

Công ty cần tích hợp dịch vụ **LDAP** (Lightweight Directory Access Protocol — giao thức quản lý danh mục người dùng) từ on-premises vào AWS VPC thông qua IAM. Identity store hiện tại **không tương thích với SAML**.

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **LDAP** = hệ thống xác thực người dùng on-premises (giống Active Directory).
2. **Không tương thích SAML** → loại bỏ các giải pháp federation chuẩn dùng SAML (như IAM Identity Center).
3. Cần giải pháp **custom** để bridge giữa LDAP on-premises và AWS credentials.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: Custom Identity Broker + AWS STS**

- Xây dựng một **custom identity broker application** on-premises — ứng dụng này:
  1. Xác thực người dùng với LDAP (hệ thống hiện có).
  2. Gọi **AWS STS** (Security Token Service) — cụ thể là `AssumeRole` hoặc `GetFederationToken`.
  3. Nhận về **temporary credentials** (access key, secret key, session token).
  4. Cấp credentials này cho người dùng để truy cập AWS resources. ✅
- Credentials là **short-lived** → an toàn hơn long-term credentials. ✅
- Giải quyết được bài toán "không tương thích SAML" vì không cần SAML. ✅

---

❌ **Đáp án sai 1: IAM policy tham chiếu LDAP identifiers**

- IAM policy không có cơ chế tích hợp trực tiếp với LDAP.
- Chỉ dùng IAM policy không đủ để thực hiện authentication và federation.
- Cần SAML, STS, hoặc custom identity broker.

---

❌ **Đáp án sai 2: AWS IAM Identity Center**

- IAM Identity Center hỗ trợ external identity providers nhưng **chủ yếu qua SAML 2.0**.
- Đề đã nói rõ "not compatible with SAML" → loại đáp án này ngay.

---

❌ **Đáp án sai 3: IAM roles để rotate credentials khi LDAP credentials thay đổi**

- Rotate credentials thủ công là **không tối ưu** và không scalable.
- Không giải quyết được bài toán tích hợp LDAP với AWS một cách có hệ thống.
- Long-term credentials kém an toàn hơn STS temporary credentials.

---

### 📊 KẾT LUẬN & BÀI HỌC

**Luồng hoạt động của Custom Identity Broker:**

```
User on-premises
      │ (nhập LDAP credentials)
      ↓
Custom Identity Broker App
      │ (xác thực với LDAP)
      ↓
LDAP Directory (on-premises)
      │ (xác nhận thành công)
      ↓
Identity Broker gọi AWS STS
(AssumeRole / GetFederationToken)
      │
      ↓
AWS STS trả về Temporary Credentials
(Access Key + Secret Key + Session Token)
      │
      ↓
User dùng Temporary Credentials → AWS Resources
```

**Khi nào dùng giải pháp nào cho federation:**

| Tình huống | Giải pháp |
|---|---|
| Identity provider **tương thích SAML 2.0** | IAM Identity Center hoặc SAML federation với STS |
| Identity provider **không tương thích SAML** | **Custom Identity Broker + STS** |
| Dùng Google, Facebook (social identity) | Amazon Cognito |
| Active Directory on-premises + SAML | AWS IAM Identity Center |

**Hai STS API hay dùng trong Identity Broker:**

| API | Dùng khi |
|---|---|
| `AssumeRole` | Broker assume một IAM role cụ thể, credentials kế thừa permission của role đó |
| `GetFederationToken` | Broker tạo federated user token với custom permissions |

💡 **Công thức ghi nhớ:**

> **"LDAP + not compatible with SAML"** → **Custom Identity Broker + AWS STS**

> **"SAML-compatible identity provider"** → IAM Identity Center hoặc SAML federation

> **STS** = Security Token Service — cung cấp **temporary credentials**, không bao giờ dùng long-term credentials cho federation

> Custom Identity Broker = giải pháp fallback khi không có SAML support — phải tự build nhưng linh hoạt nhất
