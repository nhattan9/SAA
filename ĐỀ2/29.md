## Câu hỏi 29

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company is running a multi-tier web application farm in a VPC that is not connected to their corporate network. They are connecting to the VPC over the Internet to manage the fleet of EC2 instances running in both public and private subnets. The Solutions Architect has added a bastion host with Microsoft RDP access to the application instance security groups, but the company wants to further limit administrative access to all instances in the VPC.
>
> Which bastion host deployment option will meet this requirement?

---

### 🔤 DỊCH CHI TIẾT

Công ty đang chạy multi-tier web application trong VPC không kết nối với corporate network (mạng nội bộ công ty). Họ quản lý EC2 instances qua Internet. Solutions Architect đã thêm bastion host với RDP access, nhưng công ty muốn **giới hạn thêm** quyền administrative access vào toàn bộ instances trong VPC.

Phương án triển khai bastion host nào đáp ứng yêu cầu này?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

Bài toán có 3 yêu cầu cốt lõi:

1. **Bastion host phải ở public subnet** → mới có thể truy cập từ internet vào VPC.
2. **Dùng RDP (không phải SSH)** → vì đây là Windows Bastion host (RDP = Windows, SSH = Linux).
3. **Giới hạn RDP access chỉ từ corporate IP** → không cho phép "from anywhere" → tăng cường bảo mật.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: Deploy Windows Bastion host với Elastic IP ở public subnet, chỉ cho phép RDP từ corporate IP addresses.**

- Public subnet → có Internet Gateway → có thể truy cập từ internet. ✅
- Elastic IP → địa chỉ IP tĩnh, không thay đổi → dễ quản lý trong security group rules. ✅
- Chỉ cho phép RDP từ corporate IP → giới hạn tối đa nguồn truy cập, đúng yêu cầu "further limit administrative access". ✅
- Windows Bastion → dùng RDP (port 3389) là đúng giao thức. ✅

---

❌ **Đáp án sai 1: Deploy Windows Bastion với EIP ở public subnet, cho phép SSH từ anywhere.**

- SSH (port 22) là giao thức cho **Linux**, không phải Windows. ✗
- Cho phép từ "anywhere" → không giới hạn access → vi phạm yêu cầu bảo mật. ✗
- Sai cả giao thức lẫn phạm vi truy cập. ✗

---

❌ **Đáp án sai 2: Deploy Windows Bastion trên corporate network, có RDP access đến tất cả EC2 trong VPC.**

- Bastion host phải nằm **trong VPC (public subnet)**, không phải trên corporate network. ✗
- VPC không kết nối với corporate network → bastion trên corporate network không thể reach EC2 trong VPC. ✗
- Sai về vị trí triển khai bastion host. ✗

---

❌ **Đáp án sai 3: Deploy Windows Bastion với EIP ở private subnet, giới hạn RDP chỉ từ corporate IP.**

- **Private subnet không có Internet Gateway** → không thể truy cập từ internet vào bastion. ✗
- Bastion host trong private subnet hoàn toàn không thể kết nối được từ bên ngoài VPC. ✗
- Giới hạn IP là đúng nhưng vị trí subnet sai. ✗

---

### 📊 KẾT LUẬN & BÀI HỌC

| Tiêu chí | Yêu cầu đúng |
|---|---|
| **Vị trí bastion host** | **Public subnet** (có Internet Gateway) |
| **Loại IP** | **Elastic IP** (tĩnh, dễ whitelist) |
| **Giao thức Windows** | **RDP** (port 3389) |
| **Giao thức Linux** | **SSH** (port 22) |
| **Giới hạn truy cập** | Chỉ từ **corporate IP addresses** (không phải anywhere) |

💡 **Công thức ghi nhớ:**

> Bastion host = **public subnet + EIP + security group giới hạn IP nguồn**

> **Windows** → RDP (port 3389) | **Linux** → SSH (port 22) — không bao giờ nhầm lẫn

> "Further limit access" = thu hẹp source IP trong security group, không phải mở rộng

> Bastion host trong **private subnet** = không thể truy cập từ internet = vô nghĩa với mục đích quản trị từ xa.
