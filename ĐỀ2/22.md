## Câu hỏi 22

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company hosts its web application on a set of Amazon EC2 instances in an Auto Scaling group behind an Application Load Balancer (ALB). The application has an embedded NoSQL database. As the application receives more traffic, the application becomes overloaded mainly due to database requests. The management wants to ensure that the database is eventually consistent and highly available.
>
> Which of the following options can meet the company requirements with the least operational overhead?

---

### 🔤 DỊCH CHI TIẾT

Một công ty đang host (chạy) web application trên tập hợp các Amazon EC2 instances trong một Auto Scaling group đứng sau Application Load Balancer (ALB). Ứng dụng có một embedded NoSQL database (cơ sở dữ liệu NoSQL chạy trực tiếp bên trong EC2, không tách riêng). Khi lượng traffic tăng, ứng dụng bị overloaded (quá tải) chủ yếu do database requests (các yêu cầu đến cơ sở dữ liệu). Ban quản lý muốn đảm bảo database eventually consistent (nhất quán sau cùng) và highly available (tính sẵn sàng cao).

Phương án nào đáp ứng yêu cầu với least operational overhead (ít tốn công vận hành nhất)?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

Bài toán có 3 yêu cầu cốt lõi:

1. **Giải quyết overload từ database** → cần tách database ra khỏi EC2, dùng managed service.
2. **Eventually consistent + highly available** → DynamoDB đáp ứng cả hai đặc tính này theo mặc định.
3. **Least operational overhead** → không tự quản lý replication, patching, scaling database thủ công trên EC2.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: Auto Scaling group trải đều 3 AZ + dùng AWS DMS migrate NoSQL database sang Amazon DynamoDB.**

- Trải EC2 qua 3 AZ → đảm bảo high availability cho tầng application. ✅
- AWS DMS với ongoing replication task → migrate embedded NoSQL database sang DynamoDB mà không cần downtime. ✅
- DynamoDB là fully managed service → AWS lo toàn bộ patching, scaling, replication → **least operational overhead**. ✅
- DynamoDB mặc định hỗ trợ **eventually consistent reads** và **multi-AZ high availability**. ✅

---

❌ **Đáp án sai 1: Đổi ALB sang NLB + Global Accelerator + replication NoSQL trên EC2.**

- Tiếp tục chạy NoSQL database trên EC2 → vẫn phải tự quản lý replication, scaling, patching → **operational overhead cao**. ✗
- Global Accelerator chỉ có giá trị khi kiến trúc trải rộng **nhiều AWS region** — bài toán chỉ dùng một region → thừa, không cần thiết. ✗
- NLB hoạt động ở Layer 4 (TCP/UDP), không mang lại lợi ích rõ ràng hơn ALB cho web application thông thường. ✗

---

❌ **Đáp án sai 2: Đổi ALB sang NLB + dùng AWS Glue DynamoDB export connector xuất dữ liệu sang S3 + giữ NoSQL trên EC2.**

- AWS Glue DynamoDB export connector chỉ phục vụ **analytics workload** (xuất dữ liệu để phân tích) — không giải quyết vấn đề overload của production database. ✗
- Vẫn giữ NoSQL database trên EC2 → vấn đề scalability và availability không được giải quyết. ✗
- Hoàn toàn không liên quan đến yêu cầu "eventually consistent và highly available" cho production. ✗

---

❌ **Đáp án sai 3: Auto Scaling group trải 3 AZ + tự cấu hình replication NoSQL trên EC2.**

- Về lý thuyết có thể hoạt động, nhưng yêu cầu tự cấu hình và duy trì replication database trên EC2 → **operational overhead cao**. ✗
- Không tận dụng managed service → không phải "least operational overhead". ✗
- Khi EC2 scale out/in, phải đảm bảo database node mới join/leave cluster đúng cách → phức tạp trong vận hành. ✗

---

### 📊 KẾT LUẬN & BÀI HỌC

| Tiêu chí | Tự quản lý NoSQL trên EC2 | Amazon DynamoDB (managed) |
|---|---|---|
| Operational overhead | Cao (tự patch, scale, replicate) | **Thấp (AWS quản lý toàn bộ)** |
| High availability | Phải tự cấu hình multi-AZ | **Mặc định multi-AZ** |
| Eventually consistent | Phải tự cấu hình | **Hỗ trợ sẵn** |
| Scalability | Tự scale thủ công | **Auto scaling tích hợp** |

💡 **Công thức ghi nhớ:**

> "Embedded database trên EC2 bị overload" → migrate sang **managed service (DynamoDB)** bằng **AWS DMS**.

> "Least operational overhead" → luôn ưu tiên **managed AWS service** thay vì tự vận hành trên EC2.

> "Global Accelerator" → chỉ dùng khi cần tối ưu traffic **đa region**, không phải single-region.

> "AWS Glue DynamoDB export" → chỉ dành cho **analytics/export**, không phải production database.
