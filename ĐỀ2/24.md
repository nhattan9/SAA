## Câu hỏi 24

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company plans to conduct a network security audit. The web application is hosted on an Auto Scaling group of EC2 Instances with an Application Load Balancer in front. A Solutions Architect has been tasked to enhance the security posture of the company's cloud infrastructure and minimize the impact of DDoS attacks on its resources.
>
> Which of the following is the most effective solution that should be implemented?

---

### 🔤 DỊCH CHI TIẾT

Một công ty có kế hoạch thực hiện network security audit (kiểm toán bảo mật mạng). Web application đang chạy trên Auto Scaling group các EC2 Instances với ALB phía trước. Solutions Architect được giao nhiệm vụ tăng cường security posture (tư thế bảo mật) và minimize the impact of DDoS attacks (giảm thiểu tác động của tấn công DDoS).

Giải pháp nào hiệu quả nhất cần được triển khai?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

Bài toán có 3 yêu cầu cốt lõi:

1. **Chống DDoS hiệu quả** → cần công cụ tự động chặn, không phải chỉ monitor hay notify.
2. **Lọc traffic độc hại tự động** → rate-based rules để tự động block IP khi vượt ngưỡng request.
3. **Kiến trúc đúng** → CloudFront + ALB (Layer 7), không phải NLB (Layer 4) cho web application.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: CloudFront + ALB làm origin + AWS WAF với rate-based web ACL rule.**

- CloudFront phân tán traffic toàn cầu → hấp thụ DDoS tại edge, giảm tải về origin. ✅
- ALB làm origin là đúng cho web application (Layer 7). ✅
- AWS WAF rate-based rule → tự động block IP khi số request vượt ngưỡng trong 5 phút → chặn HTTP flood attack hiệu quả, không cần can thiệp thủ công. ✅
- Có thể gắn WAF trực tiếp vào CloudFront distribution. ✅

---

❌ **Đáp án sai 1: CloudFront + NLB + GuardDuty + Lambda + SNS.**

- NLB là Layer 4 (TCP/UDP) — không phù hợp cho web application cần xử lý HTTP/HTTPS ở Layer 7. ✗
- GuardDuty là **threat detection service** (phát hiện mối đe dọa), không tự động block traffic DDoS. ✗
- Lambda + SNS chỉ gửi notification — không có cơ chế chặn tấn công. ✗
- Kiến trúc quá phức tạp, nhiều thành phần thừa. ✗

---

❌ **Đáp án sai 2: CloudFront + NLB + VPC Flow Logs + Lambda + SNS.**

- NLB không phù hợp với web application — sai về kiến trúc. ✗
- VPC Flow Logs chỉ **monitor và ghi lại** traffic patterns, không thể chặn hay mitigate DDoS. ✗
- Lambda + SNS chỉ xử lý log và gửi thông báo — vẫn không block được attack. ✗
- Giải pháp này chỉ reactive (phản ứng sau), không proactive (chủ động chặn). ✗

---

❌ **Đáp án sai 3: CloudFront + ALB + Security Group rule chặn IP đáng ngờ + SNS.**

- Security Group yêu cầu **nhập thủ công IP** cần chặn → không thể scale khi bị DDoS với hàng nghìn IP khác nhau. ✗
- DDoS attack thường dùng nhiều IP phân tán (botnet) → chặn thủ công từng IP là bất khả thi. ✗
- SNS chỉ gửi notification, không chặn traffic. ✗
- Không có cơ chế tự động → không đáp ứng yêu cầu. ✗

---

### 📊 KẾT LUẬN & BÀI HỌC

| Dịch vụ | Vai trò thực sự | Có chặn DDoS không? |
|---|---|---|
| **AWS WAF + rate-based rule** | Tự động block IP vượt ngưỡng request | ✅ Có, tự động |
| **AWS Shield** | Bảo vệ DDoS ở network/transport layer | ✅ Có (Standard miễn phí, Advanced trả phí) |
| **Amazon GuardDuty** | Phát hiện mối đe dọa, tạo findings | ❌ Chỉ detect, không block |
| **VPC Flow Logs** | Ghi lại traffic để phân tích | ❌ Chỉ monitor, không block |
| **Security Group** | Kiểm soát inbound/outbound theo rule tĩnh | ❌ Không tự động, không scale |
| **Amazon CloudFront** | CDN, hấp thụ traffic tại edge | ✅ Gián tiếp giảm tải DDoS |

💡 **Công thức ghi nhớ:**

> "Chống DDoS Layer 7 (HTTP flood)" → **AWS WAF + rate-based rule** gắn vào **CloudFront hoặc ALB**

> "Phát hiện mối đe dọa" → GuardDuty (detect only, không block)

> "Monitor traffic" → VPC Flow Logs (log only, không block)

> "Chặn IP thủ công" → Security Group (không scale với DDoS)

> Web application → luôn dùng **ALB (Layer 7)**, không phải NLB (Layer 4)
