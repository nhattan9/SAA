## Câu hỏi 30

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A solutions architect is instructed to host a website that consists of HTML, CSS, and some Javascript files. The web pages will display several high-resolution images. The website should have optimal loading times and be able to respond to high request rates.
>
> Which architecture can provide the most cost-effective and fastest loading experience?

---

### 🔤 DỊCH CHI TIẾT

Một Solutions Architect được yêu cầu host một website chỉ gồm HTML, CSS, Javascript và các ảnh độ phân giải cao. Website cần có optimal loading times (thời gian tải tối ưu) và có khả năng xử lý high request rates (lượng request lớn).

Kiến trúc nào vừa tiết kiệm chi phí nhất vừa cho tốc độ tải nhanh nhất?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

Bài toán có 3 yêu cầu cốt lõi:

1. **Website chỉ có static content** (HTML, CSS, JS, ảnh) → không cần server-side processing → không cần EC2, Beanstalk, hay web server.
2. **Tốc độ tải tối ưu + xử lý high request rate** → cần CDN (CloudFront) để cache và phân phối gần người dùng.
3. **Cost-effective nhất** → S3 static hosting rẻ hơn EC2 rất nhiều, tự động scale, không cần quản lý server.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án đúng: Upload toàn bộ HTML, CSS, JS và ảnh vào một S3 bucket, bật static website hosting, tạo CloudFront distribution trỏ đến S3 website endpoint.**

- S3 static website hosting → giải pháp tối ưu cho **pure static content**, không cần quản lý server. ✅
- Tất cả assets (HTML, CSS, JS, ảnh) trong cùng một bucket → đơn giản, dễ quản lý. ✅
- CloudFront làm CDN → cache content tại edge locations toàn cầu → tốc độ tải nhanh, latency thấp. ✅
- S3 tự động scale với high request rates → không cần cấu hình auto scaling thủ công. ✅
- **Chi phí thấp nhất**: chỉ trả theo lượng storage và request, không có chi phí EC2/compute. ✅

---

❌ **Đáp án sai 1: Auto Scaling Group với Apache web server AMI + EBS lưu ảnh + AWS Global Accelerator.**

- Dùng EC2 + ASG để host static website là **over-engineered** — tốn kém không cần thiết. ✗
- EBS không phù hợp để lưu ảnh cho website — không scale tốt, tính phí theo provision (dù dùng hay không). ✗
- Global Accelerator tối ưu cho multi-region routing, không phải CDN cache cho static content. ✗
- Đây là cách làm của thời trước khi có S3 static hosting — không còn phù hợp nữa. ✗

---

❌ **Đáp án sai 2: Host website trên AWS Elastic Beanstalk + S3 lưu ảnh + CloudFront CDN.**

- Elastic Beanstalk tạo ra toàn bộ hạ tầng (EC2, ALB, ASG) → **overkill** cho một website chỉ có static files. ✗
- Elastic Beanstalk dùng để deploy **applications có server-side logic**, không phải static HTML/CSS/JS. ✗
- Chi phí cao hơn nhiều so với S3 static hosting. ✗

---

❌ **Đáp án sai 3: Nginx server trên EC2 + S3 lưu ảnh + CloudFront CDN.**

- Tạo và quản lý Nginx server trên EC2 cho static website là giải pháp **tốn kém và thừa công sức**. ✗
- Web server (Nginx, Apache) trên EC2 phù hợp khi cần **server-side processing** (kết nối DB, xử lý logic, API). ✗
- Static website không cần server-side processing → hoàn toàn có thể dùng S3 thay thế. ✗

---

### 📊 KẾT LUẬN & BÀI HỌC

| Loại website | Giải pháp phù hợp | Chi phí |
|---|---|---|
| **Static** (HTML, CSS, JS, ảnh) | **S3 static hosting + CloudFront** | ✅ Thấp nhất |
| **Dynamic** (kết nối DB, server logic) | EC2 / Elastic Beanstalk + S3 | Cao hơn |
| **Full-stack application** | EC2 / ECS / Lambda + RDS/DynamoDB | Cao nhất |

💡 **Công thức ghi nhớ:**

> "HTML, CSS, JS, ảnh, static files" → **S3 static website hosting + CloudFront**

> "Cần server-side processing, connect to database" → EC2 hoặc Elastic Beanstalk

> S3 static hosting = **tự động scale, không quản lý server, trả phí theo dùng** → luôn là lựa chọn rẻ nhất và đơn giản nhất cho static content.

> CloudFront + S3 = **cost-effective hơn** phân phối trực tiếp từ S3 vì cache tại edge, giảm request về origin.
