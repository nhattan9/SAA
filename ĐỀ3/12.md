## Câu hỏi 12

### 📋 NỘI DUNG CÂU HỎI

> A firm has a containerized application that runs on a **self-managed Kubernetes cluster**. The cluster writes data in an **on-premises MongoDB database**. A solutions architect is requested to move the service to AWS in order to **minimize operational overhead**. The firm **prohibits any changes to the code**.

Một công ty chạy ứng dụng container trên **Kubernetes tự quản lý**, dùng **MongoDB on-premises**. Yêu cầu migrate lên AWS để giảm overhead vận hành, **không được thay đổi code**.

**Câu hỏi:** Hành động nào đáp ứng các mục tiêu này?

---

### 🔑 Từ khóa quyết định đáp án

> "Kubernetes" + "MongoDB" + "no code changes" + "minimize operational overhead"

→ Cần giữ **Kubernetes** (dùng EKS) và cần database **tương thích MongoDB** (dùng DocumentDB) để không phải đổi code.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

**✅ Đáp án đúng:** Migrate the cluster to an Amazon EKS cluster and the database to an Amazon DocumentDB (with MongoDB compatibility).

- **Amazon EKS** là Kubernetes managed service — giữ nguyên Kubernetes, giảm operational overhead, **không cần đổi code** liên quan đến container orchestration.
- **Amazon DocumentDB** tương thích với MongoDB API — ứng dụng vẫn dùng MongoDB driver như cũ, **không cần đổi code** database.
- Cả hai đều là **fully managed** — minimize operational overhead.

---

**❌ Đáp án sai 1:** Migrate to EKS Anywhere + Amazon DynamoDB.

- **DynamoDB** có API hoàn toàn khác MongoDB — phải viết lại code để thay đổi cách query, insert, update → **vi phạm điều kiện không đổi code**.
- **EKS Anywhere** dùng để chạy Kubernetes trên **on-premises** hoặc infrastructure của khách hàng, không phải trên AWS cloud — không giảm được operational overhead.

---

**❌ Đáp án sai 2:** Migrate to ECS Anywhere + Amazon Aurora Serverless.

- **Aurora** là relational database (SQL) — **không tương thích MongoDB** (NoSQL document database) → phải đổi code.
- **ECS** là container orchestration service của Amazon, **không phải Kubernetes** — migrate từ Kubernetes sang ECS có thể cần điều chỉnh.
- **ECS Anywhere** tương tự EKS Anywhere — dùng cho on-premises, không giảm overhead trên cloud.

---

**❌ Đáp án sai 3:** Migrate to ECS + ECR + Amazon Neptune.

- **Neptune** là **graph database** — dùng cho dữ liệu dạng đồ thị (mạng xã hội, knowledge graph...) — không liên quan đến MongoDB document store.
- **ECS** không phải Kubernetes — có thể cần thay đổi cấu hình.

---

### 🎯 KẾT LUẬN & BÀI HỌC

| Database | Loại | Tương thích với |
|---|---|---|
| **Amazon DocumentDB** | Document (NoSQL) | **MongoDB** ✅ |
| Amazon DynamoDB | Key-value / Document (NoSQL) | MongoDB ❌ (API khác) |
| Amazon Aurora | Relational (SQL) | MySQL / PostgreSQL |
| Amazon Neptune | Graph | Gremlin / SPARQL |
| Amazon Redshift | Data Warehouse | SQL analytics |

| Container Orchestration | Dùng khi |
|---|---|
| **Amazon EKS** | Đang dùng **Kubernetes**, muốn managed trên AWS |
| Amazon ECS | Không cần Kubernetes, muốn đơn giản hơn |
| EKS Anywhere / ECS Anywhere | Chạy trên **on-premises** hoặc infrastructure riêng |

💡 **Công thức ghi nhớ:**
> **MongoDB on-premises → AWS** = **Amazon DocumentDB** (with MongoDB compatibility) — không đổi code, không đổi driver.
>
> **Kubernetes on-premises → AWS** = **Amazon EKS** — giữ nguyên Kubernetes, AWS quản lý control plane.
>
> Khi đề có "no code changes" + tên database/framework cụ thể → tìm dịch vụ AWS **tương thích** với công nghệ đó.
