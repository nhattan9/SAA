## Câu hỏi 39

### 📋 NỘI DUNG CÂU HỎI

> A large telecommunications company needs to run **analytics against all combined log files** from the Application Load Balancer as part of **regulatory requirements**.

Công ty viễn thông cần **phân tích toàn bộ log files** từ Application Load Balancer theo yêu cầu pháp lý.

**Câu hỏi:** AWS services nào kết hợp để **thu thập logs** và **phân tích logs** dễ dàng?

---

### 🔑 Từ khóa quyết định đáp án

> "ELB log files" + "store" + "analyze" + "easily"

→ ELB logs được lưu vào **S3** (mặc định). Phân tích log lớn dễ nhất và tiết kiệm nhất trên AWS là **Amazon EMR**.

---

### 🧠 Kiến thức nền

**ELB Access Logging:**
- Mặc định **disabled** — phải bật thủ công.
- Khi bật, logs được lưu tự động vào **Amazon S3** dưới dạng compressed files.
- Không thể lưu ELB logs vào DynamoDB hay EBS trực tiếp.

**Amazon EMR (Elastic MapReduce):**
- Managed **Hadoop/Spark** framework.
- Chuyên xử lý và phân tích **dữ liệu lớn** (big data).
- Use case điển hình: **log analysis**, ETL, machine learning, web indexing.
- Đọc trực tiếp từ S3 → phân tích → xuất kết quả.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

**✅ Đáp án đúng:** Amazon S3 (store ELB logs) + Amazon EMR (analyze logs).

- **S3**: Nơi ELB tự động lưu access logs — durable, cost-effective.
- **EMR**: Tự động phân tích log lớn với Hadoop/Spark — không cần tự viết tool.
- Tiết kiệm hơn và dễ hơn so với tự build custom application.

---

**❌ Đáp án sai 1:** Amazon S3 + EC2 with custom-built application.

- Về mặt kỹ thuật đúng (S3 lưu ELB log) nhưng **tự build log analyzer trên EC2 tốn kém và mất thời gian**.
- EMR đã có sẵn framework phân tích log — không cần reinvent the wheel.
- **Không "easily perform log analysis"** như yêu cầu.

---

**❌ Đáp án sai 2:** Amazon DynamoDB + EC2.

- **DynamoDB** là NoSQL database — không phải nơi lưu ELB logs.
- ELB logs không tự động gửi vào DynamoDB.
- Lưu logs vào DynamoDB rồi dùng EC2 phân tích là cách tiếp cận **kém hiệu quả và tốn kém**.

---

**❌ Đáp án sai 3:** Amazon EC2 + EBS volumes.

- **EBS** kém bền hơn S3, không phù hợp lưu logs dài hạn.
- Tốn kém hơn — phải trả tiền EC2 + EBS 24/7 dù không phân tích.
- Không tận dụng được managed services của AWS.

---

### 🎯 KẾT LUẬN & BÀI HỌC

| Dịch vụ | Chức năng | Dùng khi |
|---|---|---|
| **Amazon S3** | Lưu ELB/ALB/NLB logs | ELB tự động gửi logs vào đây |
| **Amazon EMR** | Phân tích big data (Hadoop/Spark) | Log analysis, ETL, ML ở quy mô lớn |
| **Amazon Athena** | Query S3 data bằng SQL | Phân tích log đơn giản hơn, serverless |
| **Amazon CloudWatch** | Monitor metrics và logs real-time | Monitoring, alerting |

💡 **Công thức ghi nhớ:**
> **ELB logs → lưu vào S3** (không phải DynamoDB, không phải EBS).
>
> **Phân tích log lớn** = **Amazon EMR** (Hadoop/Spark) hoặc **Amazon Athena** (SQL query trực tiếp trên S3).
>
> EMR phù hợp cho **phân tích phức tạp, quy mô lớn**. Athena phù hợp cho **query đơn giản bằng SQL**, serverless, không cần cluster.
