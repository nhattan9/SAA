## Câu hỏi 33

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A healthcare company has developed an AWS Lambda function to handle requests from a third-party analytics service. When new patient data is available, the service sends an HTTP POST request to a webhook intended to trigger the Lambda function.
>
> What would be the MOST operationally efficient solution to ensure that the service can call the Lambda function?

---

### 🔤 DỊCH CHI TIẾT

Một công ty healthcare đã phát triển **AWS Lambda function** để xử lý request từ dịch vụ analytics bên thứ ba. Khi có dữ liệu bệnh nhân mới, dịch vụ đó gửi **HTTP POST request** đến một **webhook** (URL endpoint nhận HTTP request từ bên ngoài) để trigger Lambda function.

Giải pháp nào **operationally efficient nhất** để dịch vụ bên thứ ba có thể gọi Lambda function?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Webhook = HTTP/HTTPS endpoint public** — cần một URL có thể nhận HTTP POST request từ bên ngoài AWS.
2. **Most operationally efficient** — ít setup, ít component trung gian, ít quản lý nhất.

> ⚠️ Điểm dễ nhầm: API Gateway cũng tạo được HTTPS endpoint cho Lambda, nhưng cần **nhiều bước cấu hình hơn** → không phải "most operationally efficient".

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

---

✅ **Đáp án đúng (B):** **Generate a Lambda Function URL** và dùng làm webhook

- **Lambda Function URL** là HTTPS endpoint **chuyên dụng** được AWS tạo tự động cho Lambda function — chỉ cần vài click trong console. ✅
- Không cần setup thêm bất kỳ service trung gian nào. ✅
- Nhận HTTP POST request → trigger Lambda trực tiếp, không qua trung gian. ✅
- **Ít cấu hình nhất** trong tất cả các lựa chọn → đúng yêu cầu "most operationally efficient". ✅

---

❌ **Đáp án sai (C) — Đáp án bạn chọn:** Tạo **API Gateway endpoint** cho Lambda

- API Gateway hoàn toàn làm được ✅ nhưng cần **nhiều bước cấu hình hơn**: tạo API, tạo resource, tạo method, deploy stage, cấu hình integration với Lambda...
- Phù hợp khi cần tính năng nâng cao: authentication phức tạp, rate limiting, nhiều endpoint, request transformation...
- Với use case chỉ cần **1 webhook đơn giản** → overkill, không phải lựa chọn efficient nhất.

---

❌ **Đáp án sai (D):** **Amazon SQS queue** + dùng SQS URL làm webhook

- **SQS không chấp nhận HTTP(S) request trực tiếp từ bên ngoài AWS** → không thể dùng SQS URL như một webhook public.
- SQS là message queue nội bộ AWS, Lambda poll SQS chứ SQS không nhận HTTP request từ internet.

---

❌ **Đáp án sai (A):** **EC2 instance** làm proxy cho Lambda

- Về kỹ thuật có thể làm được, nhưng phải quản lý EC2 (patching, scaling, monitoring, cost...) → operational overhead rất cao không cần thiết.
- Hoàn toàn thừa khi đã có Lambda Function URL.

---

### Note / Pipeline đúng

Yêu cầu: Tạo webhook đơn giản để third-party service gọi Lambda qua HTTP POST, ít config nhất.

```
Third-party analytics service
→ HTTP POST request đến Lambda Function URL (HTTPS endpoint tự động)
→ Lambda Function URL trigger Lambda function trực tiếp
→ Lambda xử lý dữ liệu bệnh nhân mới
(Không cần EC2 / API Gateway / SQS)
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Giải pháp | Config effort | Dùng khi nào |
|---|---|---|
| **Lambda Function URL** | ⭐ Thấp nhất | Webhook đơn giản, 1 Lambda, ít config |
| **API Gateway + Lambda** | ⭐⭐⭐ Cao hơn | Nhiều endpoint, auth phức tạp, throttling |
| **EC2 proxy** | ⭐⭐⭐⭐ Rất cao | Không nên dùng cho use case này |
| **SQS URL** | ❌ Không khả thi | SQS không nhận HTTP request từ ngoài AWS |

💡 **Công thức ghi nhớ:**

> **"Webhook đơn giản trigger Lambda"** → **Lambda Function URL** — tạo ngay, không cần gì thêm

> **"Lambda Function URL vs API Gateway"** → URL đơn giản hơn; API Gateway mạnh hơn nhưng phức tạp hơn → chỉ dùng API Gateway khi cần tính năng nâng cao

> ⚠️ **SQS** = message queue nội bộ AWS → **không phải** HTTP endpoint public → không làm webhook được
