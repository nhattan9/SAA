## Câu hỏi 31

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A disaster recovery team is planning to back up on-premises records to a local file server share through SMB protocol. To meet the company's business continuity plan, the team must ensure that a copy of data from 48 hours ago is available for immediate access. Accessing older records with delay is tolerable.
>
> Which should the DR team implement to meet the objective with the LEAST amount of configuration effort?

---

### 🔤 DỊCH CHI TIẾT

Nhóm **DR** (Disaster Recovery — phục hồi thảm họa) muốn backup dữ liệu on-premises lên file server share qua giao thức **SMB** (Server Message Block — giao thức chia sẻ file phổ biến trên Windows). Yêu cầu:
- Dữ liệu trong **48 giờ gần nhất** phải **truy cập được ngay lập tức** (immediate access).
- Dữ liệu **cũ hơn 48 giờ** có thể truy cập chậm hơn (delay chấp nhận được).
- Giải pháp cần **least configuration effort** (ít cấu hình nhất).

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Giao thức SMB** — phải hỗ trợ SMB, không phải NFS hay giao thức khác.
2. **48 giờ gần nhất = immediate access**, dữ liệu cũ hơn = delay OK — đây là pattern **cache + S3 backend**.
3. **Least configuration effort** — ưu tiên giải pháp có sẵn, không cần setup VPN/Direct Connect.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

---

✅ **Đáp án đúng (D):** **AWS Storage File Gateway** với local cache đủ lưu 48 giờ + SMB share mount như local disk

- **File Gateway** hỗ trợ giao thức **SMB** (và NFS) — mount trực tiếp như local disk trên máy on-premises. ✅
- **Local cache** lưu dữ liệu mới nhất (48h gần nhất) → truy cập **immediate**, không độ trễ. ✅
- Dữ liệu cũ hơn lưu trên **Amazon S3** ở backend → truy cập được nhưng có độ trễ (lấy từ S3) → đúng yêu cầu. ✅
- Deploy trực tiếp on-premises, **không cần VPN hay Direct Connect** → ít cấu hình nhất. ✅

---

❌ **Đáp án sai (C):** Amazon FSx for Windows File Server + SMB share

- FSx hỗ trợ SMB ✅, nhưng FSx là service **chạy trên AWS** → để on-premises truy cập được phải **setup thêm VPN hoặc Direct Connect** → configuration effort cao hơn nhiều.

---

❌ **Đáp án sai (A):** Mount Amazon EFS + NFS share

- **EFS dùng giao thức NFS**, không phải SMB — **sai giao thức** theo yêu cầu đề bài.

---

❌ **Đáp án sai (B):** AWS Backup plan copy về local SMB share mỗi 48 giờ

- **AWS Backup chỉ hoạt động với AWS resources** (EC2, RDS, EFS...) — **không thể** backup từ on-premises hay copy về local SMB share on-premises.

---

### Note / Pipeline đúng

Yêu cầu: Backup on-premises qua SMB, 48h gần nhất truy cập ngay, dữ liệu cũ delay OK, ít config.

```
Deploy AWS Storage File Gateway on-premises
→ Cấu hình local cache đủ lớn để chứa dữ liệu 48 giờ gần nhất
→ Tạo SMB file share → mount như local disk trên máy on-premises
→ DR team copy backup vào SMB share như bình thường
→ Dữ liệu mới (≤ 48h): lưu trong local cache → truy cập immediate
→ Dữ liệu cũ (> 48h): tự động đẩy lên Amazon S3 → truy cập được nhưng có delay
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Yêu cầu / Tính năng | Service phù hợp |
|---|---|
| SMB share, on-premises, ít config | **AWS Storage File Gateway** |
| SMB share trên AWS (cần VPN/DX) | **Amazon FSx for Windows** |
| NFS share, Linux workload | **Amazon EFS** |
| Backup AWS resources | **AWS Backup** |
| Cache dữ liệu gần nhất, backend S3 | **File Gateway local cache** |

💡 **Công thức ghi nhớ:**

> **"SMB + on-premises + immediate access gần đây + delay OK cho dữ liệu cũ"** → **AWS Storage File Gateway** (local cache + S3 backend)

> **"EFS"** = NFS, không phải SMB → loại ngay khi đề yêu cầu SMB

> **"FSx for Windows"** = SMB nhưng **trên AWS** → cần thêm VPN/Direct Connect để on-premises kết nối → overhead cao hơn

> ⚠️ **AWS Backup** = chỉ cho AWS resources, **không backup on-premises**
