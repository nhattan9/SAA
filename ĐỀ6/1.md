## Câu hỏi 85

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A Solutions Architect designed a real-time data analytics system based on **Kinesis Data Stream** and **Lambda**. A week after deployment, users noticed that it performed slowly as **the data rate increases**. The Architect identified that the performance of the **Kinesis Data Streams is causing this problem**.
>
> Which of the following should the Architect do to improve performance?

---

### 🔤 DỊCH CHI TIẾT

Hệ thống phân tích dữ liệu real-time dùng **Kinesis Data Streams + Lambda**. Khi **data rate tăng** → hệ thống chậm. Đã xác định nguyên nhân là **Kinesis Data Streams**. Cần cải thiện hiệu năng.

**Từ khóa then chốt:** "data rate increases" + "Kinesis performance" → cần **tăng số shards**

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

- **Kinesis Data Streams** scale bằng cách **tăng số shards** (không phải auto-scaling hay Step Scaling)
- Mỗi shard = **1 MB/s input**, **2 MB/s output**
- Data rate tăng → cần nhiều shards hơn để xử lý
- Dùng **UpdateShardCount** API để tăng shards

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án B:** **Increase the number of shards** using the **UpdateShardCount** command.

- Tăng shards = tăng throughput của stream ✅
- **UpdateShardCount** = API chính thức để reshard Kinesis ✅
- Kinesis được thiết kế để scale **không giới hạn** bằng cách tăng shards ✅

---

❌ **Đáp án A:** Replace with **Amazon Data Firehose**.

- Firehose không có throughput cao hơn Kinesis Data Streams đáng kể ❌
- Kinesis Data Streams có thể scale không giới hạn qua shards ❌
- Thay thế toàn bộ kiến trúc là không cần thiết ❌

---

❌ **Đáp án C:** **Decrease** shards using **MergeShard** command.

- MergeShard = **gộp 2 shards thành 1** → giảm throughput ❌
- Ngược hoàn toàn với yêu cầu (cần tăng performance) ❌

---

❌ **Đáp án D:** Implement **Step Scaling** to Kinesis Data Streams.

- **Step Scaling không tồn tại** cho Kinesis Data Streams ❌
- Step Scaling chỉ áp dụng cho **EC2 Auto Scaling** ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Data rate tăng → Kinesis chậm → scale Kinesis.

```
Data rate tăng
→ Kinesis stream bị bottleneck (thiếu shards)
→ Gọi UpdateShardCount API
   (tăng số shards: VD từ 4 → 8 shards)
→ Mỗi shard xử lý 1 MB/s input / 2 MB/s output
→ Tổng throughput tăng tương ứng
→ Performance cải thiện ✅

Resharding types:
- Split shard (1 → 2): tăng capacity
- Merge shard (2 → 1): giảm capacity, tiết kiệm chi phí
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Kinesis Shard | Thông số |
|---|---|
| Input throughput | **1 MB/s** per shard |
| Output throughput | **2 MB/s** per shard |
| Tăng shards | **UpdateShardCount** (split) |
| Giảm shards | **MergeShard** (merge) |
| Auto scaling | ❌ Không có — phải manual reshard |

> 💡 **Công thức ghi nhớ:**
> "Kinesis chậm khi data tăng" → **UpdateShardCount** (tăng shards)
> "Tiết kiệm chi phí Kinesis" → **MergeShard** (giảm shards)
> **Step Scaling** = chỉ dành cho **EC2**, không có cho Kinesis
> Mỗi shard = **1 MB/s in / 2 MB/s out** — nhớ con số này!
