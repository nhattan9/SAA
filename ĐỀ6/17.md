## Câu hỏi 101

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> An application is loading hundreds of JSON documents into an Amazon S3 bucket every hour which is registered in **AWS Lake Formation** as a data catalog. The Data Analytics team uses **Amazon Athena** to run analyses on these data, but due to the volume, **most queries take a long time to complete**.
>
> What change should be made to **improve query performance** while ensuring **data security**?

---

### 🔤 DỊCH CHI TIẾT

Hàng trăm file **JSON** được upload vào S3 mỗi giờ. Dùng **AWS Lake Formation** làm data catalog, **Amazon Athena** để query. Vấn đề: **query chậm** do volume lớn. Cần cải thiện performance + đảm bảo security.

**Từ khóa then chốt:** "JSON slow queries" + "Athena performance" → **Parquet format** (columnar)

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Athena query chậm với JSON** → chuyển sang **columnar format** (Parquet hoặc ORC)
2. **Data security trong Lake Formation** → cần permission **lakeformation:GetDataAccess**
3. **GZIP compress** = tiết kiệm storage, không cải thiện query performance đáng kể

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án D:** Transform JSON → **Apache Parquet** + user có **lakeformation:GetDataAccess** IAM permission.

- **Parquet = columnar format** → Athena chỉ đọc columns cần thiết, không đọc toàn bộ file ✅
- Nhanh hơn **2x**, ít storage hơn **6x** so với JSON/CSV ✅
- Hỗ trợ **predicate pushdown** → skip blocks không cần thiết ✅
- **lakeformation:GetDataAccess** = permission bắt buộc để đọc data được kiểm soát bởi Lake Formation ✅

---

❌ **Đáp án C:** Convert JSON → **CSV** + fine-grained access control.

- **CSV = row-based format** → Athena vẫn phải đọc toàn bộ row dù chỉ cần 1 column ❌
- **Chậm hơn Parquet** vì không phải columnar format ❌

---

❌ **Đáp án A:** **Minification** + LF-TBAC.

- **Minification** (xóa whitespace JSON) = giảm nhẹ file size nhưng **không cải thiện query performance đáng kể** ❌
- LF-TBAC dùng **LF-Tags**, không phải IAM Tags thông thường ❌

---

❌ **Đáp án B:** Compress → **GZIP** + IAM policy với aws:SourceArn/aws:SourceAccount.

- GZIP **tiết kiệm storage** nhưng **không cải thiện query performance** đáng kể ❌
- aws:SourceArn/aws:SourceAccount = ngăn **confused deputy attack** — không cần thiết ở đây ❌
- **lakeformation:GetDataAccess** mới là permission đúng cho Lake Formation ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Athena query nhanh hơn + Lake Formation security.

```
JSON (row-based, chậm)
→ Convert sang Apache Parquet (columnar)
   via AWS Glue ETL job

Athena query Parquet:
→ Chỉ đọc columns cần thiết (không đọc toàn bộ) ✅
→ Predicate pushdown: skip blocks không match ✅
→ Tốn ít data scan → nhanh hơn + rẻ hơn ✅

Lake Formation security:
→ User cần: lakeformation:GetDataAccess permission
→ Lake Formation cấp temporary credentials để đọc S3 ✅
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Format | Loại | Athena Performance | Storage |
|---|---|---|---|
| **Apache Parquet** | Columnar | ✅ Nhanh nhất | 6x ít hơn JSON |
| **Apache ORC** | Columnar | ✅ Nhanh | Tốt |
| **CSV** | Row-based | ❌ Chậm | Trung bình |
| **JSON** | Row-based | ❌ Chậm nhất | Lớn |
| **GZIP** | Compression | ⚠️ Ít cải thiện | Nhỏ hơn |

> 💡 **Công thức ghi nhớ:**
> "Athena chậm" → **chuyển sang Parquet hoặc ORC** (columnar format)
> **Parquet** = 2x faster + 6x less storage so với text formats
> **lakeformation:GetDataAccess** = permission bắt buộc để đọc data qua Lake Formation
> **GZIP** = giảm storage, không giảm query time đáng kể
> **Columnar** = chỉ đọc column cần → ít I/O → nhanh hơn row-based
