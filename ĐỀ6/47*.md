## Câu hỏi 47

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company has a fleet of running Spot EC2 instances behind an Application Load Balancer. The incoming traffic comes from various users across multiple AWS regions, and you would like to have the **user's session shared** among the fleet of instances.
>
> A Solutions Architect is required to set up a **distributed session management layer** that will provide **scalable and shared data storage** for the user sessions that supports **multithreaded performance**. The cache layer must also **detect any node failures and replace the failed ones automatically**.
>
> Which of the following would be the best choice to meet the requirement while still providing **sub-millisecond latency** for the users?

---

### 🔤 DỊCH CHI TIẾT

- **fleet of Spot EC2 instances**: nhóm nhiều EC2 Spot (giá rẻ, có thể bị thu hồi bất kỳ lúc nào)
- **user's session shared**: dữ liệu phiên (session) của người dùng phải được **chia sẻ** giữa tất cả instances
- **distributed session management layer**: tầng quản lý session phân tán (không lưu trên từng EC2 riêng lẻ)
- **multithreaded performance**: hỗ trợ xử lý **đa luồng** (nhiều thread song song)
- **detect node failures and replace automatically**: tự động phát hiện node lỗi và thay thế
- **sub-millisecond latency**: độ trễ dưới 1 mili-giây (cực nhanh)

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Shared session storage**: nhiều EC2 instance cùng đọc/ghi session → cần cache tập trung
2. **Multithreaded**: hỗ trợ đa luồng
3. **Auto node failure detection & replacement**: tự động xử lý khi node chết
4. **Sub-millisecond latency**: in-memory cache (không phải database)

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

❌ **A. Amazon ElastiCache for Redis Global Datastore**

- Redis **không hỗ trợ multithreaded** theo mặc định (single-threaded)
- **Global Datastore** là tính năng cross-region replication cho DR — không phải mục đích session sharing
- Redis **không tự động detect & replace failed nodes** như yêu cầu

❌ **B. AWS ELB Sticky Sessions**

- Sticky session = gắn user vào **1 EC2 cụ thể** → session vẫn lưu local trên EC2 đó
- **Không phải shared session** — nếu EC2 đó die → session mất
- Không giải quyết được bài toán distributed session

✅ **C. Amazon ElastiCache for Memcached with Auto Discovery**

- **Memcached** hỗ trợ **multithreaded** ✅
- **Auto Discovery**: client tự động tìm tất cả nodes trong cluster, không cần hardcode endpoint
- **Tự động detect node failure và replace** ✅
- **Sub-millisecond latency** (in-memory) ✅
- Thiết kế cho **scalable, distributed caching** ✅

❌ **D. Amazon RDS database with RDS Proxy**

- RDS là **relational database** (disk-based) → latency cao hơn nhiều so với in-memory cache
- Không đạt sub-millisecond latency
- RDS Proxy giúp connection pooling, không giải quyết bài toán session caching

---

**📌 Note — Pipeline đúng:**

```
Users (nhiều region)
     ↓
Application Load Balancer
     ↓ phân phối traffic đến
Fleet of Spot EC2 Instances
     ↓ đọc/ghi session
ElastiCache for Memcached (Auto Discovery)
 ├── Node 1
 ├── Node 2
 └── Node 3
      ↕ Auto Discovery: client tự detect tất cả nodes
      ↕ Node failure → tự động replace
```

### 📊 KẾT LUẬN & BÀI HỌC

| Feature | Memcached | Redis |
|---|---|---|
| **Multithreaded** | ✅ Có | ❌ Không (single-thread) |
| **Auto Discovery** | ✅ Có | ❌ Không |
| **Auto node replacement** | ✅ Có | ❌ Không native |
| **Sub-ms latency** | ✅ Có | ✅ Có |
| **Persistence** | ❌ Không | ✅ Có |
| **Complex data types** | ❌ Không | ✅ Có |
| **Replication/Failover** | ❌ Không | ✅ Có |

> 💡 **Công thức ghi nhớ:**
> **"Multithreaded + Auto Discovery + Auto node replace → Memcached"**
> **"Persistence + Complex data + Replication → Redis"**

> 🔑 **Khi nào chọn Memcached vs Redis:**
> - Cần **scale ngang** + **multithreaded** + **simple cache** → **Memcached**
> - Cần **pub/sub**, **sorted set**, **persistence**, **failover** → **Redis**
