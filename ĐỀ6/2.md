## Câu hỏi 86

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company is running an on-premises application backed by a **1TB MySQL 8.0 database**. A couple of times each month, the production data is **fully copied to a staging database** at the request of the analytics team. The team can't work on the staging database until the copy is finished, which **takes hours**. Throughout this period, the application experiences **intermittent downtimes**. The application must be **highly resilient** to disruptions.
>
> Which combination of actions best satisfies the requirements while being the **most cost-effective**? **(Select TWO.)**

---

### 🔤 DỊCH CHI TIẾT

Công ty có MySQL database 1TB on-premises. Mỗi tháng vài lần, analytics team yêu cầu **copy toàn bộ production database sang staging** — quá trình này mất **nhiều giờ** và gây **downtime** cho ứng dụng. Cần:
1. **Copy database nhanh hơn** sang staging mà không ảnh hưởng production
2. **Highly resilient** (khả năng chịu lỗi cao)
3. **Cost-effective** (tiết kiệm chi phí)

**Từ khóa then chốt:** "hours to copy" + "intermittent downtime" + "highly resilient" → **Aurora Cloning** + **Multi-AZ**

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Copy database nhanh, không ảnh hưởng production** → **Aurora Cloning** (tính năng độc đáo của Aurora)
2. **Highly resilient** → **Aurora Multi-AZ Replicas** (tự động failover)
3. **Cost-effective** → Aurora Cloning dùng chung storage, không nhân đôi chi phí storage

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án A:** **Clone the production database** in the staging environment using **Aurora cloning**.

- **Aurora Cloning** = tạo cluster mới dùng chung Aurora cluster volume với production ✅
- **Nhanh hơn** snapshot restore hoặc mysqldump ✅
- **Không copy data vật lý** → không ảnh hưởng performance production ✅
- **Cost-effective** — chỉ lưu phần data thay đổi (copy-on-write) ✅

---

✅ **Đáp án D:** Use **Amazon Aurora database with Multi-AZ Replicas**.

- Aurora Multi-AZ = tự động replicate sang nhiều AZ ✅
- **Automatic failover** khi primary instance hỏng → highly resilient ✅
- Aurora phù hợp MySQL 8.0 (Aurora MySQL compatible) ✅

---

❌ **Đáp án B:** Use **Amazon RDS Multi-AZ Deployments**.

- RDS Multi-AZ cũng highly resilient ✅
- **Nhưng** RDS không có Aurora Cloning → vẫn phải dùng snapshot restore (chậm) ❌
- Aurora là lựa chọn tốt hơn cho use case này ❌

---

❌ **Đáp án C:** Replicate using **mysqldump** client utility.

- mysqldump = **chậm**, đây chính là vấn đề hiện tại ❌
- **Ảnh hưởng performance** của source database ❌
- Không giải quyết được vấn đề ❌

---

❌ **Đáp án E:** **Manual snapshot** + restore to staging.

- Snapshot restore **chậm hơn** Aurora Cloning ❌
- Tốn thêm storage để lưu snapshot ❌
- **Aurora Cloning nhanh và rẻ hơn** snapshot restore ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Copy database nhanh + highly resilient + cost-effective.

```
On-premises MySQL 1TB
→ Migrate lên Amazon Aurora MySQL (Multi-AZ Replicas)
   → Primary + Replicas ở nhiều AZ → highly resilient ✅
   → Automatic failover khi có sự cố ✅

Analytics team cần staging:
→ Aurora Cloning (thay vì copy toàn bộ)
   → Tạo clone cluster trong vài phút (không mất giờ) ✅
   → Dùng copy-on-write: clone dùng chung storage với production
   → Chỉ tạo copy riêng khi data thay đổi → tiết kiệm storage ✅
   → Không ảnh hưởng performance production ✅
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Phương pháp copy database | Tốc độ | Ảnh hưởng production | Chi phí |
|---|---|---|---|
| **Aurora Cloning** | ⚡ Rất nhanh | ❌ Không | Thấp (copy-on-write) |
| Snapshot restore | 🐢 Chậm | ❌ Không | Trung bình (lưu snapshot) |
| mysqldump | 🐢 Chậm nhất | ✅ Có (tốn I/O) | Thấp nhưng tốn thời gian |

> 💡 **Công thức ghi nhớ:**
> "Copy database nhanh + không ảnh hưởng production" → **Aurora Cloning**
> "Highly resilient + automatic failover" → **Aurora Multi-AZ Replicas**
> **Aurora Cloning** dùng **copy-on-write** → chỉ lưu phần thay đổi → rẻ hơn snapshot
> **mysqldump** = on-premises tool, chậm, tốn I/O → không dùng trên cloud
