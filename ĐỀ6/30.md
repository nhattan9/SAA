## Câu hỏi 30

---

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company is using an Amazon RDS for MySQL 5.6 with Multi-AZ deployment enabled and several web servers across two AWS Regions. The database is currently experiencing highly dynamic reads due to the growth of the company's website. The Solutions Architect tried to test the read performance from the secondary AWS Region and noticed a notable slowdown on the SQL queries.
>
> Which of the following options would provide a read replication latency of less than 1 second?

---

### 🔤 DỊCH CHI TIẾT

Công ty đang dùng **Amazon RDS for MySQL 5.6** với **Multi-AZ** được bật, và nhiều web server trải dài trên 2 AWS Region. Database đang chịu tải **highly dynamic reads** (đọc dữ liệu động, thay đổi liên tục) do website tăng trưởng. Khi test read performance từ Region thứ hai, phát hiện **SQL queries bị chậm đáng kể**.

Giải pháp nào đạt được **read replication latency dưới 1 giây**?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Read replication latency < 1 second** — đây là con số cụ thể, quan trọng nhất trong bài.
2. **Cross-region read** — vấn đề xảy ra khi web servers ở Region 2 đọc DB đặt ở Region 1, không phải vấn đề trong cùng Region.

> ⚠️ Điểm mấu chốt: **Multi-AZ không liên quan** đến vấn đề này — Multi-AZ chỉ dự phòng failover trong cùng Region. Vấn đề thực sự là không có bản sao DB tại Region 2. Chỉ có **Amazon Aurora replica** mới đạt latency dưới 1 giây, RDS MySQL chỉ đạt latency tính bằng giây.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

---

✅ **Đáp án đúng (D):** Migrate sang **Amazon Aurora** + tạo **cross-region read replica**

- Aurora dùng cơ chế replication đặc biệt — latency **dưới 1 giây (milliseconds)** cho cross-region replica. ✅
- Tương thích hoàn toàn với MySQL, nhanh hơn MySQL tiêu chuẩn tới 5 lần. ✅
- Web servers tại Region 2 đọc từ Aurora replica **ngay tại chỗ** — không cần vượt khoảng cách địa lý. ✅

---

❌ **Đáp án sai (A) — Đáp án bạn chọn:** Tạo **RDS for MySQL read replica** ở Region thứ hai

- Đúng hướng (cross-region read replica) nhưng **sai công nghệ** — MySQL dùng asynchronous replication, latency tính bằng **vài giây**, không đạt < 1 giây.

---

❌ **Đáp án sai (B):** Dùng **Amazon ElastiCache**

- ElastiCache là **in-memory cache** — chỉ hiệu quả khi các query **lặp lại giống nhau**.
- Bài toán là **highly dynamic reads** — dữ liệu thay đổi liên tục, query không lặp lại → cache gần như vô dụng.

---

❌ **Đáp án sai (C):** **Upgrade MySQL database engine**

- Nâng version MySQL không thay đổi cơ chế replication căn bản → vẫn asynchronous, vẫn tính bằng giây, không đạt < 1 giây.

---

### Note / Pipeline đúng

Yêu cầu: Web servers ở Region 2 đọc DB bị chậm, cần latency < 1 giây.

```
RDS MySQL ở Region 1 (web servers Region 2 đọc từ xa → chậm)
→ Migrate RDS MySQL → Amazon Aurora (MySQL-compatible)
→ Tạo Aurora cross-region read replica tại Region 2
→ Web servers Region 2 đọc từ Aurora replica ngay tại chỗ
→ Replication latency < 1 giây (milliseconds)
→ Highly dynamic reads được xử lý nhanh, không bị chậm
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Giải pháp | Replication Latency | Cross-Region | Dynamic Reads |
|---|---|---|---|
| **RDS MySQL Read Replica** | Vài giây | ✅ | ✅ |
| **Aurora Read Replica** | < 1 giây (ms) | ✅ | ✅ |
| **ElastiCache** | N/A | ✅ | ❌ chỉ tốt cho repeated queries |
| **Multi-AZ** | N/A | ❌ cùng Region | ❌ không phục vụ read |

💡 **Công thức ghi nhớ:**

> **"Latency < 1 giây / sub-second / milliseconds"** → bắt buộc dùng **Amazon Aurora replica**

> **"RDS MySQL read replica"** = latency vài **giây** → không đạt yêu cầu < 1 giây

> **"ElastiCache"** = chỉ tốt cho **repeated/static queries**, không giúp **highly dynamic reads**

> ⚠️ **Multi-AZ** = failover trong cùng Region, **không liên quan** đến read performance từ Region khác
