## Câu hỏi 108

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A global medical research company has images hosted in **S3 behind a CloudFront distribution**. When a **fresh batch of images is uploaded to S3**, it is required to **keep the previous ones** in order to **prevent them from being overwritten**.
>
> Which of the following is the most suitable solution?

---

### 🔤 DỊCH CHI TIẾT

Hệ thống upload ảnh y tế mới thường xuyên lên S3. Cần:
- **Giữ các ảnh cũ** — không bị ghi đè
- Người dùng nhận được **ảnh mới nhất** qua CloudFront

**Từ khóa then chốt:** "prevent from being overwritten" + "keep previous ones" → **Versioned objects**

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Không bị ghi đè** → cần cơ chế giữ nhiều phiên bản cùng tên file
2. **CloudFront** phục vụ đúng version mới nhất
3. **Cost-effective** → versioning rẻ hơn invalidation

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án A:** Use **versioned objects**.

- Đặt tên file theo version: `image_v1.jpg`, `image_v2.jpg`... ✅
- File cũ **không bị ghi đè** (mỗi version là file riêng biệt) ✅
- CloudFront trỏ đến version mới → không cần invalidate ✅
- **Rẻ hơn** invalidation ✅
- Kiểm soát được user nhận version nào (dù user đang cache version cũ) ✅

---

❌ **Đáp án D:** **Invalidate files** in CloudFront.

- Invalidation xóa cache CloudFront → force fetch lại từ S3 ✅ về tính năng
- **Nhưng tốn tiền hơn** versioning ❌
- Không ngăn file bị ghi đè trên S3 ❌
- User đang cache locally vẫn thấy ảnh cũ ❌

---

❌ **Đáp án C:** Add separate **cache behavior** + Minimum TTL = 0.

- TTL = 0 → CloudFront luôn check S3 mới nhất ✅
- **Nhưng không đủ** để giải quyết vấn đề gốc (file bị ghi đè) ❌
- Vẫn không kiểm soát được user cache locally ❌

---

❌ **Đáp án B:** Add **Cache-Control** headers on **S3 bucket**.

- Cache-Control nên đặt trên **từng object**, không phải toàn bộ bucket ❌
- Không giải quyết vấn đề file bị ghi đè ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Giữ ảnh cũ + phục vụ ảnh mới qua CloudFront.

```
TRƯỚC (không versioning):
Upload image.jpg (v2) → GHI ĐÈ lên image.jpg (v1) ❌
v1 bị mất!

SAU (versioning):
Upload image_v1.jpg → S3 lưu
Upload image_v2.jpg → S3 lưu (file mới, không đụng v1)
Upload image_v3.jpg → S3 lưu (file mới, không đụng v1, v2)

CloudFront trỏ đến image_v3.jpg → user nhận ảnh mới nhất ✅
image_v1.jpg và image_v2.jpg vẫn còn nguyên ✅
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Giải pháp | Giữ file cũ? | CloudFront cập nhật? | Chi phí |
|---|---|---|---|
| **Versioned objects** | ✅ | ✅ (URL mới) | Thấp |
| **Invalidation** | ❌ | ✅ | Cao hơn |
| **TTL = 0** | ❌ | ✅ | Trung bình |
| **Cache-Control trên bucket** | ❌ | ⚠️ | — |

> 💡 **Công thức ghi nhớ:**
> "Giữ file cũ + không bị ghi đè" → **Versioned objects** (đặt tên theo version)
> **Versioning** rẻ hơn **Invalidation** → AWS khuyến nghị dùng versioning
> **Invalidation** = xóa cache CloudFront, không giữ được file cũ trên S3
> Versioned objects cho phép **rollback** dễ dàng nếu version mới có lỗi
