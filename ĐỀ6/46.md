## Câu hỏi 112

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company deployed a web application to an EC2 instance that adds a variety of photo effects to a picture uploaded by the users. The application will put the generated photos to an S3 bucket by sending **PUT requests** to the S3 API.
>
> What is the best option for this scenario considering that you need to have **API credentials** to be able to send a request to the S3 API?

---

### 🔤 DỊCH CHI TIẾT

- **PUT requests to the S3 API**: yêu cầu ghi dữ liệu (upload file) lên S3 thông qua API
- **API credentials**: thông tin xác thực (Access Key ID + Secret Access Key) dùng để ký (sign) các API request
- **photo effects**: ứng dụng xử lý hiệu ứng ảnh — cần ghi kết quả lên S3

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **EC2 cần gọi S3 API**: phải có thông tin xác thực hợp lệ
2. **Bảo mật**: không được để lộ credentials (hardcode, lưu file trên server)
3. **Best practice AWS**: cách nào an toàn + dễ quản lý nhất

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **A. Create a role in IAM. Afterwards, assign this role to a new EC2 instance.**

- Tạo **IAM Role** với permission `s3:PutObject` → attach vào EC2
- EC2 tự động lấy **temporary credentials** từ Instance Metadata Service (IMDS)
- Credentials tự động **rotate** (xoay vòng) → không cần quản lý thủ công
- Đây là **AWS best practice** cho EC2 → S3 communication

❌ **B. Store your API credentials in Amazon Glacier.**

- Amazon Glacier (nay là **S3 Glacier**) là dịch vụ lưu trữ lạnh (cold storage/archive)
- Không dùng để quản lý API credentials
- Sai hoàn toàn về use case

❌ **C. Encrypt the API credentials and store in any directory of the EC2 instance.**

- Dù đã mã hóa, việc lưu credentials trên EC2 instance vẫn rủi ro
- Khó quản lý khi cần rotate credentials
- Khi instance bị compromise → credentials bị lộ

❌ **D. Store the API credentials in the root web application directory of the EC2 instance.**

- **Cực kỳ nguy hiểm** — plaintext credentials trong thư mục web app
- Dễ bị lộ qua web server misconfiguration
- Vi phạm mọi nguyên tắc bảo mật AWS

---

**📌 Note — Pipeline đúng:**

```
IAM Role (có quyền s3:PutObject)
     ↓ attach vào
EC2 Instance
     ↓ tự động lấy temporary credentials từ IMDS
     (169.254.169.254/latest/meta-data/iam/security-credentials/)
     ↓
Application gọi S3 PUT API (credentials tự động được sign)
     ↓
S3 Bucket ← ảnh sau xử lý được lưu vào đây
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Cách quản lý credentials | Bảo mật | Dễ quản lý | AWS Best Practice? |
|---|---|---|---|
| **IAM Role gắn vào EC2** | ✅ Rất cao | ✅ Tự động rotate | ✅ Có |
| Encrypt + lưu trên EC2 | ⚠️ Trung bình | ❌ Phải rotate thủ công | ❌ Không |
| Lưu plaintext trên EC2 | ❌ Rất thấp | ❌ Nguy hiểm | ❌ Không |
| Lưu trong Glacier | ❌ Sai use case | ❌ Không liên quan | ❌ Không |

> 💡 **Công thức ghi nhớ:**
> **"EC2 cần gọi AWS service → dùng IAM Role, KHÔNG dùng Access Key"**
> IAM Role = credentials tạm thời, tự rotate, không lưu trên disk

> 🔑 **Nguyên tắc vàng AWS:** Không bao giờ hardcode hoặc lưu long-term credentials trên EC2. Luôn dùng **IAM Role** → ứng dụng sẽ tự lấy credentials qua **Instance Metadata Service (IMDS)**.
