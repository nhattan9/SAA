## Câu hỏi 93

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company has an application architecture that stores both the **access key ID and the secret access key** in a **plain text file on a custom AMI**. The EC2 instances created from this AMI use the stored access keys to connect to a DynamoDB table.
>
> What should the Solutions Architect do to make the current architecture **more secure**?

---

### 🔤 DỊCH CHI TIẾT

Công ty đang lưu **access key ID** và **secret access key** (thông tin xác thực AWS) dưới dạng **plain text** (văn bản thường, không mã hóa) trong **AMI**. EC2 instances dùng key này để kết nối DynamoDB. Đây là **rủi ro bảo mật nghiêm trọng**.

**Từ khóa then chốt:** "access keys in plain text" + "AMI" → anti-pattern bảo mật → cần thay bằng **IAM Role**

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Không bao giờ hardcode access keys** vào code, file, AMI
2. **Giải pháp đúng:** dùng **IAM Role** gắn trực tiếp vào EC2 instance
3. IAM Role cấp **temporary credentials** tự động → an toàn hơn long-term keys

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án B:** **Remove** stored access keys from AMI. Create **IAM Role** with DynamoDB permissions and **assign it to EC2 instances**.

- IAM Role = **temporary credentials** tự động rotate → không cần lưu key ✅
- EC2 instance tự lấy credentials từ Role → không cần file key nào ✅
- Nếu instance bị compromise → attacker không lấy được long-term key ✅
- **Best practice AWS** cho EC2 access AWS services ✅

---

❌ **Đáp án A:** Put access keys in **Amazon Glacier**.

- Glacier là **storage service** — không phải nơi quản lý credentials ❌
- Vẫn là long-term key, chỉ chuyển chỗ lưu ❌
- Không giải quyết vấn đề bảo mật gốc rễ ❌

---

❌ **Đáp án C:** Put access keys in **Amazon S3**.

- S3 cũng là **storage service** ❌
- Vẫn là long-term key hardcoded ❌
- S3 bucket có thể bị public accidentaly → còn nguy hiểm hơn ❌

---

❌ **Đáp án D:** Do nothing — architecture is already secure.

- **Hoàn toàn sai** — plain text key trong AMI là **rủi ro bảo mật nghiêm trọng** ❌
- Bất kỳ ai có AMI đều đọc được key ❌

---

### Note / Pipeline đúng

**Yêu cầu:** EC2 cần truy cập DynamoDB một cách bảo mật.

```
TRƯỚC (không an toàn):
AMI chứa plain text: access_key_id=ABC, secret=XYZ
→ EC2 launch từ AMI → đọc file → gọi DynamoDB
❌ Key lộ nếu ai có AMI
❌ Key không tự expire
❌ Khó rotate

SAU (an toàn):
Tạo IAM Role → attach policy AmazonDynamoDBFullAccess (hoặc least privilege)
→ Gắn IAM Role vào EC2 instance
→ EC2 tự lấy temporary credentials từ Instance Metadata Service
→ Gọi DynamoDB bằng temporary credentials
✅ Không có key nào được lưu
✅ Credentials tự rotate mỗi vài giờ
✅ Nếu instance bị compromise → credentials hết hạn nhanh
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Phương pháp | Bảo mật | AWS Best Practice |
|---|---|---|
| **IAM Role trên EC2** | ✅ Cao nhất | ✅ |
| Access key trong AMI (plain text) | ❌ Nguy hiểm | ❌ |
| Access key trong S3 | ❌ Vẫn rủi ro | ❌ |
| Access key trong Glacier | ❌ Vẫn rủi ro | ❌ |

> 💡 **Công thức ghi nhớ:**
> **EC2 cần truy cập AWS service** → **IAM Role** (không bao giờ dùng access key)
> **IAM Role** = temporary credentials, tự rotate, không lưu file
> **Hardcode access key** = anti-pattern bảo mật số 1 cần tránh
>
> Mẹo nhớ: **"Role không lộ, Key có thể lộ"** — luôn dùng IAM Role cho EC2
