## Câu hỏi 79

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company has several microservices that send messages to an Amazon SQS queue and a backend application that poll the queue to process the messages. The company also has a **Service Level Agreement (SLA)** which defines the acceptable amount of time that can elapse from when messages are received until a response is sent. The backend operations are **I/O-intensive** as the number of messages is constantly growing, causing the company to **miss its SLA**. The Solutions Architect must implement a new architecture that improves the **processing time** and **load management**.
>
> Which of the following is the MOST effective solution?

---

### 🔤 DỊCH CHI TIẾT

Công ty có microservices gửi message vào **SQS queue**, backend EC2 poll queue để xử lý. Có **SLA** (Service Level Agreement — cam kết thời gian xử lý tối đa). Số lượng message tăng liên tục, backend không xử lý kịp → **vi phạm SLA**. Cần cải thiện tốc độ xử lý và quản lý tải.

**Từ khóa then chốt:** "SLA" + "time-sensitive messages" + "I/O-intensive" + "growing messages"

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Xử lý message đúng hạn SLA** → cần metric đo **tuổi của message** trong queue, không phải CPU
2. **Scale tự động** theo số lượng message → **Auto Scaling group**
3. Metric phù hợp nhất → **ApproximateAgeOfOldestMessage** (tuổi của message cũ nhất trong queue)

**Điểm dễ nhầm:** CPU metric nghe có vẻ hợp lý cho I/O-intensive, nhưng **không phản ánh thời gian message chờ trong queue** → không đảm bảo SLA.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án B:** Create AMI → Auto Scaling group → target tracking policy dựa trên **ApproximateAgeOfOldestMessage**.

- **ApproximateAgeOfOldestMessage** = đo thời gian message cũ nhất đang chờ trong queue ✅
- Khi message chờ quá lâu → metric tăng → Auto Scaling thêm instance → xử lý nhanh hơn ✅
- Trực tiếp liên quan đến **SLA** (thời gian xử lý message) ✅
- **Target tracking** = tự động scale để giữ metric ở mức target ✅

---

❌ **Đáp án D:** AMI → Auto Scaling group → target tracking policy dựa trên **CPUUtilization 80%**.

- CPUUtilization chỉ đo mức độ bận của CPU hiện tại ❌
- **Không phản ánh tuổi/thời gian chờ của message** trong queue ❌
- Instance CPU thấp nhưng message vẫn có thể chờ lâu → vi phạm SLA ❌

---

❌ **Đáp án C:** Create AMI → **replace with larger instance size**.

- Scale up (to hơn) không đủ linh hoạt để xử lý workload tăng liên tục ❌
- Không tự động điều chỉnh theo nhu cầu ❌
- Sẽ đến lúc instance lớn nhất cũng không đủ ❌

---

❌ **Đáp án A:** Create AMI → launch vào **cluster placement group**.

- **Cluster placement group** = nhóm EC2 vật lý gần nhau để giảm latency giữa các instance ❌
- Không giải quyết vấn đề scale khi message tăng ❌
- Không liên quan đến xử lý SLA ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Scale backend tự động để đảm bảo SLA xử lý message.

```
Microservices → gửi message → SQS Queue
                                   │
                    Message cũ nhất chờ N giây
                    ApproximateAgeOfOldestMessage = N
                                   │
                    CloudWatch theo dõi metric này
                                   │
              N > threshold SLA → scale out Auto Scaling group
                                   │
                    EC2 mới launch, poll thêm message
                                   │
              N giảm về mức target → scale in Auto Scaling group
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Metric | Đo gì | Dùng khi nào |
|---|---|---|
| **ApproximateAgeOfOldestMessage** | Tuổi message cũ nhất trong queue | SLA, time-sensitive messages ✅ |
| **CPUUtilization** | % CPU đang dùng | Workload CPU-bound |
| **ApproximateNumberOfMessagesVisible** | Số message đang chờ | Scale theo độ dài queue |

> 💡 **Công thức ghi nhớ:**
> "SLA" + "time-sensitive" + "SQS" → **ApproximateAgeOfOldestMessage**
> "I/O-intensive + growing messages" → **Auto Scaling group** (không phải resize instance)
> "Cluster placement group" → giảm latency giữa EC2, không phải scale theo queue
>
> Mẹo nhớ: **"Age of message" = tuổi message = trực tiếp đo SLA** → đây là metric chuẩn cho bài toán SQS + SLA
