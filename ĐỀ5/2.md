## Câu hỏi 62

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A company has a web-based ticketing service that utilizes Amazon SQS and a fleet of EC2 instances. The EC2 instances that consume messages from the SQS queue are configured to poll the queue as often as possible to keep end-to-end throughput as high as possible. The Solutions Architect noticed that polling the queue in tight loops is using unnecessary CPU cycles, resulting in increased operational costs due to empty responses.
>
> In this scenario, what should the Solutions Architect do to make the system more cost-effective?

---

### 🔤 DỊCH CHI TIẾT

Một công ty có dịch vụ bán vé (ticketing service) sử dụng **Amazon SQS** (Simple Queue Service — hàng đợi tin nhắn) và một nhóm **EC2 instances**.

Các EC2 instances đang **poll (lấy tin nhắn)** từ SQS queue liên tục nhất có thể để tối đa throughput (thông lượng). Tuy nhiên, Solutions Architect nhận thấy việc poll liên tục trong vòng lặp chặt (**tight loops**) gây lãng phí CPU cycles và tăng chi phí vận hành vì nhận về nhiều **empty responses** (phản hồi rỗng — không có tin nhắn).

**Câu hỏi:** Làm thế nào để hệ thống tiết kiệm chi phí hơn?

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

| Yêu cầu | Chi tiết |
|---|---|
| Vấn đề | Poll liên tục → nhiều empty responses → tốn CPU & tiền |
| Mục tiêu | Giảm chi phí (cost-effective) |
| Giải pháp | Thay đổi cơ chế polling của SQS |

**Từ khóa bẫy:** "tight loops", "empty responses", "unnecessary CPU cycles" → đây là dấu hiệu của **Short Polling** (mặc định) đang gây vấn đề → cần chuyển sang **Long Polling**.

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **A. Configure Amazon SQS to use long polling by setting the ReceiveMessageWaitTimeSeconds to a number greater than zero.**
→ **ĐÚNG.** Long Polling = SQS đợi cho đến khi có tin nhắn mới trả về → giảm empty responses → giảm chi phí. `ReceiveMessageWaitTimeSeconds > 0` (tối đa 20 giây) = bật Long Polling.

❌ **B. Configure Amazon SQS to use short polling by setting the ReceiveMessageWaitTimeSeconds to zero.**
→ SAI. `ReceiveMessageWaitTimeSeconds = 0` chính là cấu hình **Short Polling mặc định** — đây là nguyên nhân gây ra vấn đề, không phải giải pháp.

❌ **C. Configure Amazon SQS to use short polling by setting the ReceiveMessageWaitTimeSeconds to a number greater than zero.**
→ SAI kép: `ReceiveMessageWaitTimeSeconds > 0` là **Long Polling**, không phải Short Polling → nhầm tên lẫn giá trị.

❌ **D. Configure Amazon SQS to use long polling by setting the ReceiveMessageWaitTimeSeconds to zero.**
→ SAI kép: `ReceiveMessageWaitTimeSeconds = 0` là **Short Polling** → nhầm tên nhưng đúng giá trị của Short Polling.

---

### 🗺️ NOTE / PIPELINE

```
EC2 poll SQS (Short Polling mặc định)
→ Query chỉ một số server ngẫu nhiên
→ Có thể trả về rỗng dù vẫn còn tin nhắn
→ EC2 lại poll ngay → vòng lặp liên tục
→ Tốn CPU + tăng API call cost

✅ Giải pháp: Bật Long Polling
→ ReceiveMessageWaitTimeSeconds = 1~20 giây
→ SQS đợi tối đa N giây cho đến khi có tin nhắn
→ Query TẤT CẢ server (không bỏ sót)
→ Giảm số lần gọi API → giảm chi phí
```

---

### 📊 KẾT LUẬN & BÀI HỌC

**So sánh Short Polling vs Long Polling:**

| Đặc điểm | Short Polling | Long Polling |
|---|---|---|
| `ReceiveMessageWaitTimeSeconds` | = 0 (default) | > 0 (tối đa 20s) |
| Query servers | Một phần (subset) | Toàn bộ |
| Empty responses | Nhiều | Ít/không có |
| Chi phí API | Cao hơn | Thấp hơn |
| Phù hợp | Cần throughput cao tức thì | Cần tiết kiệm chi phí |

> 💡 **Công thức ghi nhớ:**
> **ReceiveMessageWaitTimeSeconds = 0** → **Short Polling** (mặc định, tốn tiền)
> **ReceiveMessageWaitTimeSeconds > 0** → **Long Polling** (tiết kiệm, giảm empty responses)
>
> Mẹo nhớ: **"Chờ lâu hơn (> 0) = Long = Tiết kiệm"**
