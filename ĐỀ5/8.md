## Câu hỏi 70

### 📋 NỘI DUNG CÂU HỎI (Tiếng Anh gốc)

> A social media company needs to capture the detailed information of all HTTP requests that went through its public-facing Application Load Balancer every five minutes. The **client's IP address** and **network latencies** must also be tracked, while the load balancer's availability is monitored separately through the ELB health check. The captured data should be used for **analyzing traffic patterns** and for **troubleshooting Docker applications** orchestrated by the **Amazon ECS Anywhere** service.
>
> Which of the following options meets the customer requirements with the **LEAST amount of overhead**?

---

### 🔤 DỊCH CHI TIẾT

Công ty mạng xã hội cần:
- Capture (bắt) **toàn bộ thông tin chi tiết HTTP requests** qua ALB mỗi 5 phút
- Theo dõi **client IP address** (địa chỉ IP client) và **network latencies** (độ trễ mạng)
- Phân tích **traffic patterns** (mẫu lưu lượng truy cập)
- Troubleshoot (gỡ lỗi) Docker apps chạy trên **Amazon ECS Anywhere** (ECS mở rộng ra ngoài AWS, chạy trên on-premises hoặc bất kỳ cloud nào)
- Yêu cầu: **ít overhead nhất** (ít công sức vận hành nhất)

---

### 🎯 PHÂN TÍCH YÊU CẦU CHÍNH

1. **Capture HTTP request chi tiết (client IP, latency)** → tính năng **ALB Access Logs** — ghi lại mọi request vào S3
2. **Phân tích traffic + troubleshoot ECS apps** → **CloudWatch Application Insights** — tự động phân tích, tạo dashboard
3. **Least overhead** → loại bỏ giải pháp cần cài đặt agent thủ công, cấu hình phức tạp

**Từ khóa bẫy:** "client IP address" → X-Ray không capture IP, CloudTrail không capture HTTP traffic

---

### 🔍 PHÂN TÍCH TỪNG ĐÁP ÁN

✅ **Đáp án C:** Enable **access logs** on ALB. Integrate ECS cluster with **Amazon CloudWatch Application Insights**.

- **ALB Access Logs** ghi lại: thời gian request, **client IP**, **latency**, request path, server response → đúng yêu cầu ✅
- Logs lưu vào **S3** tự động (mỗi 5 phút) → không cần cấu hình phức tạp ✅
- **CloudWatch Application Insights** tự động scan tài nguyên, cấu hình metrics/logs, tạo dashboard → troubleshoot ECS apps ✅
- Powered by SageMaker → phát hiện anomaly tự động → **least overhead** ✅

---

❌ **Đáp án B:** Enable **AWS CloudTrail** for ALB. Use **CloudTrail Lake** to analyze traffic.

- **CloudTrail** chỉ ghi lại **API calls / resource changes** đến ALB (ai tạo, ai sửa cấu hình...) ❌
- **Không capture HTTP traffic thực tế** (client IP, request path, latency) đi qua ALB ❌
- CloudTrail Lake dùng cho SQL query trên API events, không phải application traffic ❌

---

❌ **Đáp án A:** Install **AWS X-Ray daemon** on ECS. Use **CloudWatch ServiceLens**.

- X-Ray trace request qua microservices nhưng **không capture client IP address** ❌
- Bài yêu cầu track client IP → thiếu tính năng quan trọng ❌
- Phải cài X-Ray daemon thủ công → overhead cao hơn ❌

---

❌ **Đáp án D:** Integrate **Amazon EventBridge metrics** on ALB. Use **CloudWatch GetMetricData**.

- **EventBridge không capture HTTP request detail** như client IP ❌
- **CloudWatch GetMetricData** chỉ lấy metrics đã có sẵn (aggregate), không có request-level data ❌
- Không đáp ứng yêu cầu capture client IP và latency chi tiết ❌

---

### Note / Pipeline đúng

**Yêu cầu:** Capture HTTP detail (IP + latency) + analyze traffic + troubleshoot ECS, least overhead.

```
HTTP requests → ALB
→ ALB Access Logs bật (tự động, không cần agent)
   → Ghi: client IP, latency, path, response code...
   → Lưu vào S3 mỗi 5 phút ✅

ECS Anywhere cluster
→ Tích hợp CloudWatch Application Insights
   → Tự động scan resources
   → Tự động cấu hình metrics & logs
   → Tạo dashboard, phát hiện lỗi tự động ✅
```

---

### 📊 KẾT LUẬN & BÀI HỌC

| Yêu cầu | Giải pháp |
|---|---|
| Capture HTTP request detail (IP, latency) | **ALB Access Logs** → lưu vào S3 |
| Phân tích traffic patterns | **ALB Access Logs** + Athena query |
| Troubleshoot ECS apps, least overhead | **CloudWatch Application Insights** |
| Ghi lại API calls thay đổi config ALB | CloudTrail (không phải HTTP traffic) |
| Trace request qua microservices | X-Ray (không có client IP) |

> 💡 **Công thức ghi nhớ:**
> "Client IP + HTTP detail + latency" → **ALB Access Logs** (bật là xong, lưu S3)
> "Troubleshoot app, least overhead" → **CloudWatch Application Insights** (tự động hóa)
> "CloudTrail" → chỉ track **API changes**, không track **HTTP traffic**
> "X-Ray" → trace microservices, **không có client IP**
>
> Mẹo nhớ: **ALB Access Logs = hộp đen của load balancer** — ghi lại mọi request đi qua
